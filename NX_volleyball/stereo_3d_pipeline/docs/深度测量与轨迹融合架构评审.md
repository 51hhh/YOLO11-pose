> 评审对象: `stereo_3d_pipeline` 的双路 YOLO → 多候选深度 → 混合深度 → 轨迹预测链路
> 评审分支: `debug-feature-match-viz`
> 评审日期: 2026-07-02
> 评审依据: 源码实读 + `test_logs` 探针实测数据 + NX 实机 profiler 复测，非文档转述。凡标注"推断"处均为未实测。

> 后续落地状态(2026-07-02): 两路彩色候选和 `roi_neural_feature` 已接入实时候选体系。ORB 已接到 true OpenCV CUDA ORB,但 NX 同步实测 78-79 FPS 且 0/512 有效;BRISK/AKAZE/SIFT 在当前 NX 无 true CUDA 后端,历史 GPU-lite 只能作为自研快速候选/对照。神经路径已移除 `matchGpuRoi` stub,支持 fused TensorRT engine;XFeat extractor-only TensorRT 已实测,但后处理仍回 CPU 且同步路径不能 100fps。最新真实 GPU 复核见 `test_logs/nx_true_gpu_features_20260702/report.md`。
> 后续落地状态(2026-07-03): 检测帧 `stage2_roi_match_fuse()` 已整段异步化，worker 使用自有 gray/BGR/host 快照 buffer，下一帧 YOLO ready 后过期旧 ROI 后处理结果。神经 matcher 已按 engine schema 区分 gray 与 BGR 输入，避免 3/6 通道模型误读灰度内存。

## 0. 结论速览

当前架构**整体合理，方向正确**，尤其是"多候选深度并行计算 + 强几何 gate 剔除 + 数据落盘为未来融合做准备"这条主线判断得很准。但存在四个需要调整的结构性问题：

| # | 问题 | 严重度 | 类型 |
|---|------|--------|------|
| A | 多候选深度在线只做**硬编码优先级择一**，没有融合，十几路候选算了却只用一路 | 高 | 架构 |
| B | GPU 稀疏特征 `corner/texture/binary` 可同时运行，但**全开实测 9.48ms，不能进 100fps 默认路径** | 高 | 性能 |
| C | `confidence` 数值与 gate 通过与否**语义不一致**，直接当权重会被"点多但散"骗到 | 中 | 正确性 |
| D | 轨迹层只有 `predictOnly` 纯 Kalman 外推 + 独立 `TrajectoryPredictor`，**未利用多候选的可靠性信息** | 中 | 架构 |

下面逐条展开，并给出对应的四个原始问题的评审。

## 1. 实际数据流（核实后）

```
PWM 100Hz 硬触发
  ↓
双路 YOLO 并发   pipeline.cpp:4044-4103
  detector_ (左, detect_stream) + detector_right_ (右, detect_stream_right)
  两个独立 TRTDetector 实例 + 两条独立 CUDA stream → 真并发
  ↓
IoU 语义配对   epipolar_y_tolerance=12px 的 y 门 + size_ratio 门
  ↓
每个配对 → 一批候选深度
  GPU: dualYoloDepthCandidatesKernel  一次 launch 批量算
       (圆拟合/边缘质心/径向中心/边缘对中心/center-patch/multi-point/corner/texture/binary
        + 历史 ORB/BRISK/AKAZE/SIFT-lite + 彩色 BGR 候选)
  OpenCV CUDA: true ORB A/B 路径
  CPU: OpenCV BRISK/AKAZE 仅作 debug/A-B + 亚像素兜底
  ↓
depth_source 优先级链择一   pipeline.cpp:3358-3493   ← 关键：择优不是融合
  ↓
HybridDepthEstimator   只吃 roi_results[i].z 一个标量   hybrid_depth.cpp:320
  mono/stereo IVW 融合 + 9D 恒加速 Kalman
  其余十几路候选仅透传进 CSV   hybrid_depth.cpp:400-495
  ↓
ROS2 /ball/realtime + landing 预测 (main.cpp 的 TrajectoryPredictor)
```

**必须澄清的三个事实**（后面分析都基于此）：

1. **在线是"择优"不是"融合"**。`pipeline.cpp:3358` 起是一条长 if-else 链，按固定优先级选**第一个 valid 的候选**作为 disparity，其余候选不参与最终深度。
2. **HybridDepth 只接收一路标量 z**。`roi_results[i].z` 是上一步择优的结果；十几路 `z_*` 候选在 `hybrid_depth.cpp:400-495` 只是被拷进 `Object3D` 供 CSV 记录，**不参与任何在线计算**。
3. **轨迹预测器在 pipeline 外**。`TrajectoryPredictor`（弹道 RK4 + 多项式）只在 `main.cpp:767` 调用；pipeline 内部丢帧时只用 `hybrid_depth_->predictOnly()` 做纯 Kalman 外推。

## 2. 问题一评审：各特征点提取算法的 GPU 加速与实时耗时

### 2.1 架构评价：GPU 批处理设计优秀

`dual_yolo_depth_gpu.cu`（1753 行）的调度模型是这套系统最扎实的部分：

- **一个检测配对 = 一个 CUDA block**（`pair_idx = blockIdx.x`），最多 64 对，`kThreads=128`。
- **特征点匹配用 `kThreadsPerPoint=4`** 协作搜一个点的视差范围，`128/4=32` 点并行。
- **几何法 + center-patch + corner/texture/binary 全部在同一 block 内、单次 kernel launch 完成**，用 shared memory（`edge_x/y/w`、`sample_*`、`best_*_parts`）承载中间量。

这个设计的关键收益：多种候选共享同一次调度和数据搬运，避免了每个方法单独 launch 的固定开销。**但实测证明，`corner/texture/binary` 这类稀疏特征的 block 内循环并不便宜，不能把"同一 kernel"等同于"边际成本可忽略"。**

### 2.2 NX 实机 profiler：默认几何路径很快，稀疏特征不能常开

2026-07-02 在 NX `nvidia@10.42.0.149` 复测，使用 `codex_stereo_3d_pipeline_review` 评审副本 + 当前相机/engine。默认 100fps 配置中 `roi_corner_points/roi_texture_points/roi_binary_points=false`：

| 配置 | `Stage2_DualYoloGpuCandidates` | `Stage2_DualYoloMatch` | 输出表现 |
|---|---:|---:|---|
| 默认几何候选 | avg **0.28ms**, min 0.24, max 0.96 | avg 0.30ms | ROI FPS 约 100 |
| 仅 `roi_corner_points=true` | avg **3.45ms** | avg 3.47ms | FPS 约 40-65 |
| 仅 `roi_texture_points=true` | avg **2.45ms** | avg 2.47ms | FPS 约 64-78 |
| 仅 `roi_binary_points=true` | avg **3.46ms** | avg 3.48ms | FPS 约 72-78 |
| 三者全开 | avg **9.48ms**, max 22.05 | avg 9.51ms | FPS 约 16-34，出现 stale drop |

结论很明确：

- 默认几何候选路径可以进 100fps 主链路，GPU wrapper 端到端约 0.28ms。
- `corner/texture/binary` 能跑，也能同时跑，但只能作为离线/低频/A-B 测试项；三者全开会直接吞掉单帧预算。
- profiler 统计的是 C++ wrapper 端到端（含 H2D/kernel/D2H/sync），不是纯 kernel 时间，因此对实时预算更有参考价值。

### 2.3 CPU 侧成本控制（已做对）

- `corner/texture/binary` 走 GPU；CPU 版 `matchSparseFeatureDisparityCPU` 仅 GPU 缺失兜底。
- **ORB 已有 true OpenCV CUDA 实时候选,BRISK/AKAZE/SIFT 仍无 true CUDA 后端**，且默认全关（`roi_orb/brisk/akaze/sift_points: false`）。OpenCV CUDA ORB 对当前排球场景连续 512 帧有效候选为 0,同步平均 3.24ms；BRISK/AKAZE/SIFT 只能按 CPU debug 或历史 lite 对照处理。默认关掉仍是对的。
- 亚像素有硬预算闸 `subpixel_time_budget_ms=1.5`（`pipeline.cpp:3342`），超预算 `subpixel_budget_skip`。这是防 CPU 拖垮帧率的正确设计。

### 2.4 结论与行动项

**评审结论**：GPU 批处理架构合理，默认几何候选路径足够快；但稀疏特征候选不是"免费"的，必须保持默认关闭，并按单项 profiler 预算决定是否低频启用。CPU 成本控制方向正确。

**行动项 P0**：
1. 100fps 配置继续保持 `roi_corner_points/roi_texture_points/roi_binary_points=false`，只开几何/圆/边缘类候选。
2. 若要启用稀疏特征，必须单项开关 A/B，并同时记录 `Stage2_DualYoloGpuCandidates`、`Stage2_DualYoloMatch`、ROI FPS 和 `stale_drop_count`。
3. 将 profiler 日志纳入 `test_logs` 或 benchmark 产物，避免性能结论再次停留在口头复测。

## 3. 问题二评审：各特征提取方式能否同时运行

**评审结论：功能上能，且已实现；性能上不能把所有 mode 当默认常开。**

- `depth_modes` 是一组独立开关（`pipeline_dual_yolo_roi.yaml:55-73`），每个 mode 独立产出 `z_*` + `disparity_*` + `support/std/confidence` 并写 CSV。
- GPU 侧几何/patch/corner/texture/binary 同 kernel 并行，互不冲突；但 2.2 的 NX 实测显示，`corner/texture/binary` 三者全开会把 `Stage2_DualYoloGpuCandidates` 拉到 9.48ms。
- 所有候选共享同一 `initial_disp` 先验和同一套 gate。
- `recording.detail_level: extended` + `raw_mode: true` 已把这十几路候选全部落盘（`schema.md` 定义了完整字段表）。

这套"并行多方法 + 宽表落盘"正是为问题四的模型训练准备的数据基础，设计连贯。

**唯一提醒**：同时开全部 mode 时，GPU 候选 kernel 的 shared memory 占用、寄存器压力和匹配搜索循环都会上升。当前 100fps 默认配置应保持稀疏特征关闭，只把它们用于离线质量对比或低频诊断。

## 4. 问题三评审：匹配效果与误匹配剔除

### 4.1 剔除链设计：扎实，对症

每个候选点串联通过多层 gate（`roi_feature_match_cpu.cpp`，GPU kernel 对称实现）：

1. **极线 y 残差门** `strictFeatureYTolerance`（clamp 0.5–8px，配置 2px）
2. **重叠椭圆门** `passesFeatureOverlapGate`——点须在左右检测椭圆内，且用视差先验投影到对侧后仍在椭圆内
3. **球体半径门** `passesSphereRadiusGate`——三角化 3D 点须在物理球体积附近（半径 0.10m × 1.8 + 2cm）。**这是针对光滑球面的强约束，是本系统的亮点**
4. **左右互检** `reverseSparseMatchError`（误差超 `feature_reverse_check_px=1.0` 剔除）
5. **唯一性**（best-second 分差）
6. **视差 delta 门** `computeSubpixelDispDeltaGateCPU`（绝对 px / 比例 / 深度米数三者取最紧）
7. **RANSAC + MAD 一致性** `aggregateRobustMatches`（中值 + 1.4826×MAD×scale，加权中值视差，最后 stddev 门）

层次分明，覆盖了误匹配的主要来源。设计质量高。

### 4.2 实测数据（`offline_keypoint_probe_recommended`，~3.4m，基线1m，视差~440px）

| 方法 | 匹配数 | z_mad | 置信度 | 状态 | 失败原因 |
|---|---|---|---|---|---|
| **iou_region_color_patch** | 18 | **7.7mm** | 0.65 | ✅ pass | — |
| patch_iou_zncc_corner | 53 | 15.6mm | 0.93 | ❌ | y_error/disp_mad/z_range |
| akaze | 8 | 4.2mm | 0.24 | ❌ | valid_points<8 |
| orb | 4 | 1.4mm | 0.24 | ❌ | valid_points<8 |
| brisk | 4 | 6.2mm | 0.16 | ❌ | valid_points<8 |
| sift | 1 | — | 0.08 | ❌ | valid_points<8 |

参数扫描：324 组里 242 组 pass，最佳 `mask_margin=10, patch_radius=9, min_score=0.54, y_radius=1`（z_mad 7.7mm, valid_ratio 1.0）。

### 4.3 三个关键发现

1. **描述子类（ORB/BRISK/AKAZE/SIFT）在光滑排球上物理失效**——纹理太少，keypoint 只有个位数，全部 `valid_points<8`。默认关闭正确，不是性能问题而是适用性问题。

2. **高 confidence ≠ 高质量（问题C的核心证据）**。`patch_iou_zncc_corner` 拿到 78 点、confidence 0.93，却因 y_error/disp_mad 失败，valid_ratio 仅 26%。**confidence 公式（support+score+consistency 加权，`roi_feature_match_cpu.cpp:1014`）会被"点多但散"的情况骗到；真正把关的是 gate**。这直接影响问题四的融合权重设计——**不能拿 confidence 当逆方差权重**，应该用 gate 通过后的 `stddev/support`。

3. **赢家是边缘/patch 类 + 强 gate**（`iou_region_color_patch`）。对光滑球面，边缘/圆几何比稀疏特征点更可靠。

### 4.4 需要注意的隐患

- **大视差近距场景（~440px）误匹配空间大**。`max_disparity=2048`，搜索范围宽，目前主要靠 sphere gate 兜住。远距小视差场景（视差几~几十 px）的剔除效果**没有实测数据**，而那恰恰是双目最需要、也最容易量化误差爆炸的区间——建议专门补一组远距探针。
- **神经特征匹配不再是固定空壳,但还没有真实模型闭环验证**。`matchGpuRoi` 已实现 fused TensorRT engine 入口、ROI crop/resize、`[N,4]/[N,5]` 匹配输出解析和 gate,pipeline 会生成 `roi_neural_feature` 候选;无 engine path 时 warn 后禁用。XFeat/ALIKED/SuperPoint 真实 engine 导出、split extractor/matcher 运行时和质量/FPS 对拍仍待完成。

## 5. 问题四评审：用模型融合多深度值 + 历史时序拟合真实轨迹

### 5.1 方案形态：可行，且 `trajectory_fusion/README.md` 的判断正确

推荐结构（README 已定，我完全认同）：

```
多路观测/质量特征/历史序列
  → MeasurementReliabilityNet (GRU)   输出每方法 log_sigma / bias / outlier_prob
  → 鲁棒 EKF/UKF (物理模型)
  → 最终 x/y/z/v/a 轨迹 + 落点预测
```

**不做端到端黑盒/GAN 是对的**：
- 无真值轨迹标签 → GAN 判别器只会学"看起来平滑"，不保证深度绝对正确。
- 100FPS 闭环要可解释、可限幅、可回退 → 生成器失效不可控。
- 噪声本质（bbox 抖动/视差误匹配/远距量化/单侧漏检/同步异常）**天然是观测噪声+偏差+离群概率模型**。

`models.py` 的 `MeasurementReliabilityNet` 骨架对口：GRU 编码时序，输出 log_sigma/bias/outlier_logit；`losses.py` 用 Student-t/Huber NLL + 留一法 + 物理约束（x''≈0, z''≈0, y''≈g）+ 不确定度校准。设计成熟。

### 5.2 落地的两个真实障碍

**障碍1：训练数据字段不够（当前最大瓶颈，非模型问题）。**
`trajectory_fusion/README.md` 已诚实指出：现有 `raw_observation_data.csv` 只是**在线 Kalman 输出**，不是原始多路观测。离线 smoother 在其上只能降高阶抖动（z_std 0.2307→0.2312 几乎不动）。好消息是 `extended` + `raw_mode` 已在写 `schema.md` 定义的候选深度 + 质量特征列，**缺的是用新格式重采一批覆盖各场景的数据**。

`schema.md` 里训练诊断字段的当前状态：
- 已补进 recorder：`predicted_z` / `innovation_z` / `innovation_norm` / `kalman_sigma_z`
- 仍缺：`rejected_reason`（每个候选被拒原因）和逐帧曝光/增益

没有"被拒候选 + 失败原因"，模型学不到"什么时候不该信谁"。

**障碍2：在线是择优不是融合（问题A）。** 接入点很明确——把 `pipeline.cpp:3358` 的 if-else 链替换成"收集所有 valid 候选 → reliability net → 鲁棒加权融合"。`models.py` 的 `weighted_depth_consensus` 已写好融合函数。

### 5.3 落地路线建议

1. **先补数据，再谈模型**：用 `extended/raw_mode` 采覆盖 静止 / 飞行 / 单侧漏检 / 远近距 的数据集。重点补**远距小视差**和**动态飞行**（近距 3-4m 探针已证明很稳）。同时补齐 5.2 的诊断字段。
2. **离线建 baseline**：先跑 `robust_smoother.py`（纯物理鲁棒平滑，无需训练），再训 `MeasurementReliabilityNet`，用留一法确认"学会了信谁"。
3. **只把轻量部分接回在线**：`TinyCausalFusionHead`（无循环，吃已算好的时序 delta 特征，目标 <0.3ms）适合在线；GRU 版留离线。
4. **物理滤波器兜底**：reliability net 只出噪声/偏差/离群概率，最终轨迹仍由 EKF 保证物理一致，可限幅、可回退到当前 HybridDepth。

## 6. 现有在线融合与轨迹层的具体问题（补充）

### 6.1 HybridDepth 的 mono/stereo 融合（`hybrid_depth.cpp`）

**合理的部分**：
- 9D 恒加速 Kalman 手写 3×3 块展开，避免通用 9×9 乘法，性能好。
- 距离自适应观测噪声 `R(z)=R_base·z²`，符合三角测距误差随距离平方增长的物理。
- IVW 逆方差加权在过渡带 [stereo_min_z, mono_max_z] 线性混合，`stereo_bias_` EMA 自适应校正 zs/zm 比例——设计细致。

**存疑的部分**：
- **方法切换用 `z_pred`（上一帧 Kalman z）判断**（`hybrid_depth.cpp:339`），初始化时 `z_pred=z_mono`。若单目在近距偏差大，会影响首帧方法选择，可能造成冷启动抖动。
- **`stereo_bias_` 全局单值**，多目标场景（虽然当前是单排球）会串扰。单球场景没问题，但要在注释里标明这个假设。
- `getObsNoise` 的 blend 方差推导（`hybrid_depth.cpp:186-195`）用 IVW 权重的平方加权，与 Kalman 的 R 是两套独立参数（`R_mono/R_stereo` vs `ivw_R_mono/ivw_R_stereo`），**四个噪声参数需要一起标定**，否则容易顾此失彼。

### 6.2 depth_source 优先级链（`pipeline.cpp:3358-3493`）

优先级顺序（valid 即选）：
```
subpixel(2) → feature_fallback(12) → template_fallback(7) → circle/fallback(1)
→ center_patch(4) → corner(10) → texture(11) → binary(13) → orb(14) → brisk(15)
→ akaze(16) → radial(8) → edge_pair(9) → edge_centroid(5) → bbox_center(3) → bbox_edge(6)
```

**问题**：
- 顺序是**硬编码经验排序**，不随场景/质量动态调整。例如 subpixel 永远优先，但 4.2 数据显示 subpixel（=multi_point）在光滑球面未必比边缘/圆几何稳。
- 一旦高优先级方法"valid 但质量差"（回到问题C：valid≠准），低优先级的好候选没机会。
- 这条链本质就是**问题A**在代码里的体现——它是"融合缺位"的临时替代品。

**建议**：短期可在链里加入 stddev/gate 作为 valid 的更严判据（而非只看 `.valid` flag）；长期用 5.2 的 reliability 融合替换整条链。

### 6.3 轨迹预测器

- `TrajectoryPredictor`（弹道 RK4 + 多项式 fallback）只在 `main.cpp` 跑，pipeline 内部丢帧只用 `predictOnly()` 纯外推——两者**没有共享历史缓冲**，落点预测的输入是 Kalman 平滑后的单点，没利用多候选可靠性。
- `ground_y` 默认在 `.h` 里是 2.5m 但配置文件 `pipeline_dual_yolo_roi.yaml` 是 0.0——**默认值与实机配置不一致**，需确认落点预测用的是哪个，避免落点算错。

## 8. 总体判断

架构没有方向性错误，核心链路（双路并发 YOLO → GPU 批量多候选 → 强几何 gate → Kalman → 物理预测）是一套自洽、工程化程度高的实时系统。真正需要补的是**两个缺口**而非重构：

1. **性能治理缺口**：已有 NX profiler 证明默认几何候选可达 100fps，但稀疏特征全开会严重超预算；后续必须把 profiler 产物常态化归档。
2. **融合缺口**：多候选算了却只择一路用，`trajectory_fusion` 的设计正确但卡在"原始多路观测数据没采够"。

先补数据和性能归档，再上 reliability 融合模型——顺序不能反。在补齐 `schema.md` 原始观测前，任何绝对深度精度结论只能做趋势判断。
