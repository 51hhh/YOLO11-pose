## 4. 关键发现

### 4.1 DLA回退机制的精度保护效应

**最重要的发现**: DLA无法执行attention块(MatMul)，回退到GPU以FP16运行。这反而**保护了精度**：

| 推理路径 | Attention精度 | 其余层精度 | 检测率 |
|----------|-------------|-----------|--------|
| GPU INT8 | INT8 ⚠️ | INT8 | 65% |
| DLA INT8 | **FP16 (GPU回退)** ✅ | INT8 (DLA) | 88% |
| DLA FP16 | **FP16 (GPU回退)** ✅ | FP16 (DLA) | 98% |

- GPU INT8将attention块也量化为INT8，丢失23%精度
- DLA INT8的attention在GPU上以FP16运行，仅丢失10%精度
- **结论: attention块对INT8量化极度敏感**

### 4.2 模型对比

| 指标 | v11dla | v11attn | yolo26 | v10n |
|------|--------|---------|--------|------|
| GPU INT8延迟 | 3.12ms ⭐ | 3.58ms | 3.20ms | ❌全败 |
| GPU INT8精度 | 65% | 65% | 86% ⭐ | ❌ |
| DLA INT8延迟 | 8.13ms ⭐ | 10.44ms | 17.11ms | ❌ |
| DLA INT8精度 | 88% ⭐ | 87% | 87% | ❌ |
| DLA FP16精度 | 98% | 98% | 99% ⭐ | ❌ |
| DLA FP16延迟 | 23.10ms ⭐ | 25.10ms | 28.68ms | ❌ |
| DLA覆盖率 | 高 | 中 | 低 | 极低 |

- **v11dla在所有DLA配置中延迟最低** (DLA层覆盖率最高)
- **yolo26在GPU INT8中精度最高** (86% vs 65%，INT8更友好)
- **yolo26在DLA FP16中精度最高** (99%)
- **v10n完全不兼容当前TRT版本**

## 5. 推荐方案

### ~~方案A: GPU FP16~~ ❌ 已排除
> **8GB workspace重试全部失败**
- v11dla/v11attn/yolo26/v10n 全部 FAIL (254-258s)
- 原因: TRT 10.3.0对这些模型的attention MatMul层**没有FP16 GPU实现**
- 这不是workspace问题，是算子级别的兼容性限制
- **结论: GPU FP16路径在TRT 10.3.0上对含attention的YOLO模型完全不可用**

### 方案B: v11dla DLA0 INT8 + 双DLA流水线
> **当前最佳折中方案**
- 配置: DLA0+DLA1交替执行v11dla_dla_int8_entropy2引擎
- 延迟: 8.13ms (单帧) → 流水线吞吐量翻倍
- 精度: 88% (接近90%目标，差2%)
- GPU释放: DLA处理大部分计算，GPU仅处理attention回退
- 配置文件:
  ```yaml
  detector:
    engine_path: "matrix_results/v11dla_dla0_int8_entropy2.engine"
    engine_path_dla1: "matrix_results/v11dla_dla1_int8_entropy2.engine"
    dual_dla: true
    confidence_threshold: 0.25  # 降低阈值可能提升检测率
  ```

### 方案C: yolo26 GPU INT8 (最佳速度方案)
> **延迟优先时选择**
- 延迟: 3.20ms ✅ (远低于7.5ms目标)
- 精度: 86% (仅差4%)
- GPU占用: 100% (无DLA释放)

### 方案D: 自适应混合策略
> **工程化方案，需要更多开发**
- 正常帧: GPU INT8 (~3ms, 65-86%精度)
- 关键帧(每N帧): DLA FP16 (~23ms, 98%精度)
- 平均延迟: 取决于关键帧比例
- 实现: 需要修改pipeline逻辑

## 6. 最终结论

### 硬约束

在 TRT 10.3.0 + Orin NX 平台上：
1. **GPU FP16 全不可用** — attention MatMul 无实现（4GB和8GB workspace均失败）
2. **MinMax校准** — trtexec不支持，需Python API
3. **v10n** — 完全不兼容
4. **无任何单引擎配置同时满足 >90%精度 + <7.5ms延迟**

### 推荐配置（按优先级）

| 优先级 | 配置 | 精度 | 延迟 | 适用场景 |
|--------|------|------|------|---------|
| ⭐1 | v11dla DLA0+DLA1 INT8 双DLA流水线 | 88% | 8.13ms→吞吐翻倍 | GPU释放，接近目标 |
| ⭐2 | yolo26 GPU INT8 (entropy2) | 86% | 3.20ms | 延迟优先，精度可接受 |
| ⭐3 | v11dla DLA FP16 + 双DLA交替 | 98% | 23ms→吞吐翻倍 | 精度优先 |
| ⭐4 | 混合策略: GPU INT8常规帧 + DLA FP16关键帧 | ~90%+ | ~5ms avg | 需开发自适应逻辑 |

### 突破方向
1. **导出不含attention的YOLO模型** → 可能解锁GPU FP16
2. **升级TRT 10.5+** → 可能新增attention MatMul FP16支持
3. **混合精度引擎** (`--layerPrecisions`) → 指定attention层FP16+其余INT8
4. **Python API构建** → 手动指定每层精度，绕过trtexec限制

### A. 混合精度引擎测试（2026-04-05 新增）

#### A.1 突破：`--layerPrecisions` 混合精度构建成功

使用 TRT `--precisionConstraints=prefer --layerPrecisions="*attn*":fp16 --layerOutputTypes="*attn*":fp16` 参数，
成功构建了 **attention层FP16 + 其余层INT8** 的混合精度引擎。

**构建方法**: 
```bash
trtexec --onnx=model.onnx --int8 --fp16 \
  --calib=model_calib_entropy2.cache \
  --precisionConstraints=prefer \
  --layerPrecisions="*attn*":fp16 \
  --layerOutputTypes="*attn*":fp16 \
  --saveEngine=mixed_attn_fp16.engine
```

#### A.5 Pipeline 端到端对比（2026-04-05 新增）

在完整 4-Stage ROI Pipeline 中测试三种部署模式（真实海康双目相机 100Hz 触发，各运行15秒）：

| 配置 | 引擎 | 帧数 | FPS | DetectSubmit | WaitDetect | WaitRect | 瓶颈 |
|------|------|------|-----|-------------|-----------|----------|------|
| **GPU Mixed** | yolo26_mixed_attn_fp16 | 1354 | **~90** | 3.54ms | 0.01ms | 5.20ms | VPI Rectify |
| **Dual DLA** | yolo26_dla0/1_fp16 | 487 | **~33** | 3.00ms | 14.89ms | 9.27ms | DLA推理23ms |
| **Triple** | DLA0+DLA1+GPU Mixed | 683 | **~45** | 2.93ms | 11.92ms | 2.22ms | DLA推理 |

**Pipeline 阶段说明**:
- `Stage0_GrabRect`: 相机抓帧 + VPI 校正 (PVA)
- `Stage1_DetectSubmit/WaitRect`: TRT 推理提交 + 等待校正完成
- `Stage2_WaitDetect/ROIMatchFuse`: 等待推理 + ROI立体匹配 + 3D融合
- `ROIMatchFuse`: ROI深度估算 (focal=1485.5, baseline=0.3287m, 单/双目混合)

**模式选择建议**:
- **最低延迟**: GPU Mixed —— 90 FPS, 端到端 <10ms, 检测仅 3.5ms
- **释放 GPU**: Dual DLA —— 33 FPS, GPU 空闲可并行其他任务
- **吞吐平衡**: Triple —— 45 FPS, WaitRect 降至 2.2ms (GPU 只做 1/3 检测)

> **注**: WaitRect 在 Dual DLA 模式下从 5.2ms 升至 9.27ms，原因是 DLA attention 前16层 fallback 到 GPU
> 与 VPI Rectify 竞争 GPU 带宽。Triple 模式中 GPU 仅承担 1/3 帧的完整推理，其余时间
> 全部服务 Rectify，WaitRect 降至 2.22ms。
