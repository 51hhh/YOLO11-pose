## 实时 GPU 候选

主要文件:

- `src/stereo/dual_yolo_depth_gpu.h`
- `src/stereo/dual_yolo_depth_gpu.cpp`
- `src/stereo/dual_yolo_depth_gpu.cu`

入口类:

```cpp
stereo3d::DualYoloDepthGpuMatcher
```

输入:

- 校正灰度左右 CUDA 指针。
- 可选校正 BGR 左右 CUDA 指针。
- 左右 YOLO bbox pair。
- 相机焦距、基线、主点和深度范围。

输出:

```cpp
std::vector<DualYoloGpuCandidate>
```

当前候选:

|---|---|
| 几何/圆 | `left_circle`, `right_circle`, edge/radial/edge-pair point |
| 中心 patch | `center_patch` |

| sparse-lite | `corner_points`, `texture_points`, `binary_points` |
| 历史描述子近似 | `orb_points`, `brisk_points`, `akaze_points`, `sift_points` lite kernel 仍存在，但当前 pipeline 不再把它们当真实算法启用 |

## 后端限制核对

资料和源码边界:

- OpenCV 官方 CUDA features2d 暴露 `cv::cuda::ORB`，当前代码在 `src/stereo/roi_feature_match_gpu.cpp` 使用它和 CUDA `DescriptorMatcher`，这是唯一已接入实时 C++ 的真实 OpenCV CUDA 传统描述子。
- OpenCV CPU `features2d` 暴露 `cv::BRISK`、`cv::AKAZE`、`cv::SIFT`；当前项目没有对应 `cv::cuda::BRISK` / `cv::cuda::AKAZE` / `cv::cuda::SIFT` 后端，也没有自研完整 CUDA 版本。这三者在实时管线中只能走 OpenCV CPU，或作为历史 sparse-lite 对照字段，不能写成真实 GPU 实现。
- VPI 官方文档明确提供 CUDA Template Matching、CUDA Stereo Disparity、CUDA Harris、CUDA Pyramidal LK、ORB feature detector/descriptor 和 BruteForceMatcher。当前已接入 VPI Template/Stereo、VPI Harris/LK、VPI ORB diagnostic-only 真实后端；10:49 targeted 复核中 Template 为 `601/630` 有效但有长尾，Harris/LK 为 `48/606`，ORB 为 `16/603`，均未达到默认实时准入。
- Fixstars libSGM 是公开 CUDA SGM 实现，README 给出 Xavier MAXN 约 `9ms` 的全幅 benchmark；当前项目已接 `roi_libsgm` diagnostic wrapper、CMake 可选依赖检测和 `scripts/nx_build_libsgm.sh`。NX 已可构建并链接 `third_party/libSGM/install`，08:30 isolated 实测为 `70.5fps`、`0/80` 有效且存在 `5.2s` 级长尾，不准入。
- XFeat、ALIKED、SuperPoint、LightGlue 的实时 C++ 路径只接受 TensorRT engine。Python/Torch probe 只用于离线语义验证，不允许进入生产进程。
- 神经 matcher 当前支持三类真实 TensorRT schema: XFeat extractor-only、direct extractor `keypoints/descriptors/scores`、fused 点对输出；split LightGlue matcher 只支持固定 keypoint/descriptor 输入和 `matches`/`matches0` 输出 schema。未知动态 schema 会被标记 unsupported，不会假装成功。

## P2 加速路线

P2 只收真实 GPU/TensorRT 后端已接入、或已经明确要 isolated 测试的实现路线；历史 CPU 调试项、旧混跑项和本轮明确暂停的第三方实现不再放入 P2。

| 算法/方向 | GPU 化方式 | 当前动作 |
|---|---|---|
| 颜色 IoU/patch | 已在自研 CUDA batch 中实现 | 增加离线推荐参数和宽搜索参数 case，后续单项复测 |
| OpenCV CUDA ORB | `cv::cuda::ORB` + CUDA BF matcher | 增加 `fast48` 和宽 y gate case，判断耗时是否由 ROI/点数/门限导致 |
| 自研 CUDA Template/NCC | 自研小 ROI CCOEFF_NORMED/NCC kernel + GPU peak reduce | 当前默认 `z_roi_cuda_template_match` 后端；OpenCV CUDA TemplateMatching 仅保留 `STEREO_CUDA_TEMPLATE_BACKEND=opencv` baseline |
| OpenCV CUDA StereoBM/SGM | `cv::cuda::StereoBM` / `cv::cuda::StereoSGM` | 已接入局部 ROI dense disparity，增加 `patch9` case |
| XFeat | TensorRT extractor，后续优先把 NMS/descriptor sampling/mutual-NN 推到 GPU 或导出 fused 点对 engine | 增加 `96/128/160/224` ROI 和 top-k sweep |
| SuperPoint | fixed TensorRT extractor，后续优先低 ROI/top-k 或 fused SuperPoint+LightGlue | 增加 `128/160/224 top64` sweep |
| ALIKED | 需要无 DCN 导出或 TensorRT deformable convolution plugin | 保留 `160/224 top64` case；缺 engine 时脚本跳过 |
| VPI Template/Stereo | VPI CUDA backend | 已接真实 VPI CUDA diagnostic-only 后端并通过 NX build；10:49 VPI Template 为 `601/630` 有效但有 `57.54ms` 长尾，只保留 diagnostic；VPI Stereo 仍按历史无有效/不准入口径处理 |
| VPI Harris/LK、VPI ORB | VPI CUDA backend | 已接真实 diagnostic-only 后端并通过 NX build；10:49 Harris/LK `48/606`、VPI ORB `16/603`，均不准入；VPI ORB 已有算法级点对 artifact |
| OpenCV CUDA GFTT/LK、CUDA HoughCircles | OpenCV CUDA backend | 已接真实 OpenCV CUDA diagnostic-only 后端并通过 NX build；有球实测均不准入 |
| libSGM / SGM-lite | 第三方 CUDA SGM 或自研小 ROI cost volume | Fixstars libSGM 已接 wrapper 和 build 脚本，08:30 isolated 已实测不准入；自研 SGM-lite 仍是未落地后续方向 |

## 不再使用/不再推进

| 方法 | 当前处理 | 原因 |
|---|---|---|
| OpenCV CPU ORB/BRISK/AKAZE/SIFT | 只保留 debug/offline，不进入默认实时和 P2 结论 | 需要 CPU 图像，历史测试有效率低或超预算 |
| OpenCV 官方 BRISK/AKAZE/SIFT CUDA 路线 | 本轮不再推进 | 当前没有可用的 `cv::cuda::BRISK` / `cv::cuda::AKAZE` / `cv::cuda::SIFT` |
| CUDA-SIFT 第三方实现 | `roi_cuda_sift` 只保留 unsupported diagnostic/config 占位 | 用户已决定标准 CUDA-SIFT 暂不做，不引入第三方依赖 |
| `all_sparse_gpu`、`corner/texture/binary` sparse-lite | 不再作为正式实时算法矩阵项 | 它们是历史轻量近似特征，不能写成真实 ORB/BRISK/AKAZE/SIFT |

## 实时输入和内存需求

不同方法不能只按“算法名字”归类；实时开销主要由输入格式、计算后端和是否需要 CPU 可读图像决定。

| 方法/类别 | 图像输入 | 主要计算后端 | CPU 图像/CPU 后处理 | 关键限制 |
|---|---|---|---|---|
| 默认几何、edge/radial/edge-pair | gray GPU | CUDA | 无 | 100fps 主路径 |
| `roi_center_patch`, `roi_subpixel`, `multi_point` | gray GPU | CUDA + 少量 CPU gate | 无 CPU 图像 | async 后 `roi_subpixel/multi_point` 满足 10ms, 但需评估深度偏置 |
| `corner/texture/binary` sparse-lite | gray GPU | 自研 CUDA sparse-lite | 无 CPU 图像 | 不是 OpenCV 描述子算法；连续帧有效率见实时矩阵 |
| 彩色 IoU/edge patch | BGR GPU | CUDA | 无 CPU 图像 | 需要 BGR 管线和 async BGR snapshot |
| 自研 CUDA Template/NCC | gray GPU | 自研 CUDA CCOEFF_NORMED/NCC + peak reduce | 不需要 CPU 图像；默认只回传最终 peak，debug 时才下载 score patch | P2 默认关闭，只在直接左右 YOLO pair 上运行；OpenCV CUDA 仅作 baseline |
| OpenCV CUDA StereoBM | gray GPU | `cv::cuda::StereoBM` | 不需要 CPU 图像；下载小 ROI disparity 做 robust 聚合 | P2 默认关闭；08:30 patch9 无有效候选，不准入 |
| OpenCV CUDA StereoSGM | gray GPU | `cv::cuda::StereoSGM` | 不需要 CPU 图像；下载小 ROI disparity 做 robust 聚合 | P2 默认关闭；10:49 patch9 `139/624` 有效且 max 过 10ms，不准入 |
| OpenCV CUDA ORB | gray GPU | `cv::cuda::ORB` + CUDA matcher | 不需要 CPU 图像；keypoints/matches 元数据会回 CPU 做 gate；`gpu_candidate_refine=true` 时不为 CPU ORB fallback 预先 D2H | 真 GPU ORB；strict gate 有效率低，relaxed 有效率可上升但仍约 11ms 且尾延迟大 |
| OpenCV BRISK/AKAZE/SIFT | gray pinned host | CPU OpenCV | 必需 CPU 图像 | 当前 NX 无 true CUDA 后端，默认关闭 |
| XFeat extractor-only | 1ch gray 或 3ch BGR GPU ROI | TensorRT extractor | 不需要 CPU 图像；网络输出 D2H 后做 NMS、descriptor 采样和 mutual-NN | 后处理仍 CPU，不是完整 GPU 常驻匹配 |
| ALIKED/SuperPoint direct extractor | 1ch gray 或 3ch BGR GPU ROI | TensorRT extractor | 不需要 CPU 图像；输出 D2H 后做 descriptor mutual-NN/gate | 需要真实 engine 输出 `keypoints/descriptors/scores` |
| TensorRT split matcher | 1ch gray 或 3ch BGR GPU ROI | TensorRT extractor + TensorRT matcher | 不需要 CPU 图像；当前中间特征会 D2H/H2D，最终 gate 在 CPU | 仅 `use_lightglue=true` 时启用；matcher 只支持固定 keypoint/descriptor 输入和 `matches`/`matches0` 输出 schema |
| SuperPoint+LightGlue fused | 2ch gray-pair 或 6ch BGR-pair GPU ROI，也支持两个 1/3ch 输入 | TensorRT fused matcher | 不需要 CPU 图像；输出点对 D2H 后 CPU gate | 需要 fused engine 输出 `[N,4]` 或 `[N,5]` |
| fallback template/feature | gray pinned host | CPU fallback | 必需 CPU 图像 | 仅漏检/回退场景使用，不应作为 100fps 默认重路径 |

在 `performance.async_roi_stage2=true` 时，async buffer 资源池按配置预分配，但每帧 snapshot 按当帧检测结果决定是否复制:

- gray GPU 是完整 Stage2 的基础输入，检测帧提交 async ROI 时会复制到 worker 自有 buffer。
- BGR GPU 只在当帧左右目有配对机会，且彩色候选或 3/6 通道神经 engine 可能运行时复制。
- pinned host gray 只在配置启用 CPU OpenCV BRISK/AKAZE/SIFT，或当帧单侧检测、多检测框歧义、无法预配对导致 CPU fallback 可能运行时复制；正常一左一右 GPU-only 配对帧不做 D2H。
- `roi_orb_points` 单独启用时走真实 OpenCV CUDA ORB，不触发 full-frame host gray 复制；CPU ORB fallback 只会在关闭 GPU refine，或其他 CPU 路径已经提供 host gray 时发生。
- CPU OpenCV BRISK/AKAZE/SIFT 不是 GPU 实时算法；打开这些开关会触发 host gray 路径，应只用于 A/B 或 debug。

## NX GPU 实时流水线

当前实时路径的调度在 `src/pipeline/pipeline_loops.cpp`，检测帧异步提交和 worker 在 `src/pipeline/pipeline_async_roi.cpp`，核心匹配仍由 `src/pipeline/pipeline.cpp` 中的 `stage2_roi_match_fuse()` / `runRoiStage2Core()` / `matchDualYoloDetections()` 驱动:

```text
100Hz 同步帧
  -> VPI/CUDA 校正, FrameSlot 缓存 rectGray/rectBGR CUDA 指针
  -> 左右 YOLO TensorRT detect stream
  -> Stage2 等待 detect event 并 collect 检测
  -> IoU + 极线/尺寸 gate 配对左右 bbox
  -> DualYoloDepthGpuMatcher 一次 CUDA batch 生成几何/patch/sparse-lite/颜色候选
  -> 可选自研 CUDA Template/NCC、OpenCV CUDA StereoBM/StereoSGM/ORB、CPU OpenCV BRISK/AKAZE/SIFT、TensorRT XFeat/ALIKED/SuperPoint/LightGlue
  -> CPU 端 gate/候选整理/legacy P0/fallback 兼容选择/记录 CSV
```

关键 profile:

| profile | 含义 |
|---|---|
| `Stage2_DualYoloGpuCandidates` | `dual_yolo_depth_gpu.cu` 批量候选耗时 |
| `Stage2_CudaTemplateNccMatch` | 自研 CUDA Template/NCC 小 ROI 极线匹配耗时 |
| `Stage2_OpenCVCudaStereoBM` | OpenCV CUDA StereoBM 小 ROI dense disparity 耗时 |
| `Stage2_OpenCVCudaStereoSGM` | OpenCV CUDA StereoSGM 小 ROI dense disparity 耗时 |
| `Stage2_CudaRingEdgeProfileMatch` | 自研 CUDA ring/edge profile 小范围极线匹配耗时 |
| `Stage2_P2FeatureJobDiagnosticCudaRingEdgeProfile` | diagnostic lane ring/edge profile 耗时 |
| `Stage2_P2FeatureJobDiagnosticVpiTemplate` | VPI CUDA TemplateMatching diagnostic 耗时 |
| `Stage2_P2FeatureJobDiagnosticVpiStereo` | VPI CUDA StereoDisparity diagnostic 耗时 |
| `Stage2_P2FeatureJobDiagnosticVpiHarrisLk` | VPI CUDA Harris + Pyramidal LK diagnostic 耗时 |
| `Stage2_P2FeatureJobDiagnosticVpiOrb` | VPI CUDA ORB + BruteForceMatcher diagnostic 耗时 |
| `Stage2_P2FeatureJobDiagnosticFixstarsLibSgm` | Fixstars libSGM diagnostic 耗时 |
| `Stage2_P2FeatureJobDiagnosticOpenCVCudaGfttLk` | OpenCV CUDA GFTT/Harris + SparsePyrLK diagnostic 耗时 |
| `Stage2_P2FeatureJobDiagnosticCudaHoughCircle` | CUDA HoughCircles circle refinement diagnostic 耗时 |
| `Stage2_OpenCVCudaORB` | true OpenCV CUDA ORB ROI 提取和匹配耗时 |
| `Stage2_NeuralFeatureMatch` | TensorRT 神经特征 ROI 推理和当前 CPU 后处理耗时 |
| `Stage2_CPUFeatureSparse*` | CPU sparse corner/texture/binary fallback 耗时 |
| `Stage2_CPUFeatureOpenCV*` | CPU OpenCV ORB/BRISK/AKAZE/SIFT 耗时 |

| `Stage2_DualYoloMatch` | 左右配对、候选读取、gate 和深度候选整理 |
| `Stage2_ROIMatchFuse` | Stage2 ROI 候选生成、legacy 兼容选择和输出总耗时 |

当前重要边界:

- 默认几何候选在 GPU 上足够快, 是 100fps 主路径。
- `roi_cuda_template_match` 是自研 CUDA Template/NCC P2 候选，OpenCV CUDA TemplateMatching 只作为 baseline 后端保留；`roi_cuda_stereo_bm`、`roi_cuda_stereo_sgm` 是独立 OpenCV CUDA P2 候选。`roi_ring_edge_profile` 是自研 CUDA ring/edge profile P2 候选。它们写原始 `z_*` 字段，不参与 legacy `z_stereo`/`obj.z` 选择，也不需要 host gray。
- OpenCV CUDA ORB 虽然是真 GPU 算法, 但同步路径会阻塞 Stage2；当前已新增整段异步 Stage2，超出下一帧 YOLO-ready deadline 的结果会被丢弃。
- XFeat extractor 在 TensorRT 上很快, 但当前左右各跑一次 extractor, 并把 `feats/keypoints/heatmap` 拷回 CPU 做 NMS、descriptor 采样和互反查匹配。
- ALIKED/SuperPoint direct extractor、split LightGlue matcher 和 fused matcher 都已有 C++ TensorRT 入口；可用性取决于真实 engine 的固定 shape 和 tensor name/schema。SuperPoint fixed extractor 已在 NX 闭环但超预算；ALIKED 当前无可用 TensorRT engine。
- BRISK/AKAZE/SIFT 在当前项目中是 CPU OpenCV 路径；async Stage2 只能把它们移出主调度线程并按下一帧 YOLO-ready deadline 丢弃过期结果，不能把它们变成 GPU 算法。
- `performance.async_roi_stage2=true` 时，重特征不再阻塞主线程继续取流和提交下一帧检测；但 running CUDA/CPU work 不能强杀，只能完成后按 frame deadline 丢弃结果。
- 后续若要进一步降低 GPU 争用和 snapshot copy 成本，应把重特征从整段 Stage2 中拆成小 ROI `FeatureJob`，使用独立 stream/context 和独立 ROI buffer。

## 真实 OpenCV CUDA ORB

主要文件:

- `src/stereo/roi_feature_match_cpu.h`
- `src/stereo/roi_feature_match_cpu.cpp`
- `src/stereo/roi_feature_match_gpu.cpp`
- `src/pipeline/pipeline.cpp`
- `src/pipeline/pipeline_async_roi.cpp`

当前状态:

- `roi_orb_points` 在有校正灰度 CUDA 图像时调用 `matchOpenCVORBDisparityGPU()`。
- 该路径使用 `cv::cuda::ORB::create()` 和 CUDA `DescriptorMatcher`，是真实 OpenCV CUDA ORB，不是 lite patch 近似。
- 外部 pitched ROI 会先复制到连续 `cv::cuda::GpuMat`，这是为了避开 OpenCV CUDA ORB 对非连续 ROI 的非法访存风险。
- ORB detect/describe 和 BF matcher 在 CUDA 上执行；`convert()` / `knnMatchConvert()` 会把 keypoints 和 match 元数据回 CPU，后续几何 gate 和 robust 聚合仍在 CPU。
- `gpu_candidate_refine=true` 时，`roi_orb_points` 不再为 CPU ORB fallback 预先复制 full-frame host gray；如果 GPU ORB 无效且没有其他 CPU 路径提供 host gray，本帧 ORB 候选直接无效。
- 结果仍通过 y 残差、重叠椭圆、球体半径、反查、视差 delta 和 MAD 聚合 gate。

资料核对和 NX 实机结论:

- OpenCV 文档暴露 `cv::cuda::ORB` 和 CUDA `DescriptorMatcher`，因此当前 OpenCV CUDA ORB 是真实算法实现。
- 当前项目没有 BRISK、AKAZE、SIFT 的等价 CUDA 后端；这些字段只能走 CPU OpenCV 或历史 lite 对照，不能写成真实 GPU 实时实现。
- VPI ORB 已按 VPI3 CUDA ORB feature detector + BRIEF descriptor + BruteForceMatcher 接入 diagnostic-only 后端；10:49 targeted 为 `95.9fps`、`16/603` 有效、algo p95 `9.60ms`，有效率不足且每帧运行仍不准入；10:53 debug run 已输出真实左右点对 artifact。
- 2026-07-03 遮挡后 clean rerun 中，OpenCV CUDA ORB strict gate 连续测试 `27/399` 有效，`Stage2_AsyncRoiWorker` 平均 `11.02ms`、最大 `50.50ms`、over-deadline `152`，输出约 `75.5fps`。relaxed gate 提升到 `249/389` 有效，但平均仍 `10.96ms`、最大 `44.99ms`、over-deadline `168`。它可作为真实算法 A/B，但不是 100fps 默认方案。

## CPU 调试候选

主要文件:

- `src/stereo/roi_feature_match_cpu.h`
- `src/stereo/roi_geometry_cpu.h`
- `src/stereo/roi_patch_match_cpu.h`

用途:

- 生成 debug feature match 图片。
- 离线验证 sparse/OpenCV 特征是否能在真实排球上稳定匹配。
- 在无 GPU 或专项 debug 时作为对照。

本地 CPU / 离线路径:

| 路径 | 用途 | 注意 |
|---|---|---|
| `--debug-feature-matches` | NX 或本机单帧抓取并输出 legacy CPU 匹配图 | 用于看 sparse/OpenCV CPU 点位和失败原因，不是 P2 GPU/VPI/TRT 后端内部可视化，也不是连续帧性能测试 |
| `tools/offline_volleyball_keypoint_probe.py` | 本地 CPU 传统特征、IoU、patch 诊断 | 输出 summary 和 triangulated points, 适合调 gate |
| `tools/neural_feature_probe.py` | 本地 XFeat/ALIKED/SuperPoint+LightGlue 正确性验证 | PyTorch/CPU/GPU 结果不能等同 NX TensorRT 实时性能 |
| `tools/compare_offline_online_depth.py` | 离线结果和在线 CSV 字段对拍 | 用于确认实时近似是否偏离离线语义 |

本地 CPU 结论:

- OpenCV ORB/BRISK/AKAZE/SIFT 在光滑排球上通常点数少, 连续帧稳定性不足。
- 离线神经特征能证明几何上可匹配, 但进入 NX 实时路径前必须导出 TensorRT engine 并实测。
- 本地 CPU 结果只作为语义和 gate 对照, 不能直接作为 100fps 能力证明。

当前不放进 100fps 默认路径:

- OpenCV BRISK/AKAZE 在当前 NX 上没有真实 CUDA 后端；OpenCV ORB 虽有 CUDA 后端并已有点对 artifact，但实测 FPS/有效率仍不准入。
- CPU 特征和图片可视化会引入不可忽略的 CPU 时间和内存复制。
- 单帧看起来正确不代表连续帧稳定。

这些限制只约束当前实现，不阻止同名算法或替代算法以真实 GPU 后端重新进入 P2 isolated matrix。

## 神经特征匹配

配置文件:

- `src/stereo/neural_feature_config.h`
- `src/stereo/neural_feature_matcher.*`

支持 backend:

| backend | 说明 |
|---|---|
| `xfeat` | 轻量局部特征 |
| `aliked` | 轻量关键点和描述子 |
| `superpoint_lightglue` | SuperPoint + LightGlue 路径 |

当前 C++ 实时路径重点是 fused TensorRT engine:

- 支持 GPU ROI crop/resize。
- 支持一个输入张量 C=2 或 C=6，也支持两个输入张量。
- 输出最后一维 4 或 5 时解释为左右点对和可选 score。
- 执行 positive disparity、max disparity、y error、initial disparity delta、MAD/final gate、min matches 等过滤。

当前 XFeat extractor-only split 路径也已有 C++ fallback:

- 条件: `backend=xfeat`，`extractor_engine_path` 有效，且没有可用 fused engine。
- 输入: 固定 ROI TensorRT extractor；C=1 使用 gray，C=3 必须使用 BGR 管线和校正 BGR 指针，输出 `feats/keypoints/heatmap`。
- 后处理: extractor 输出通过 `cudaMemcpyAsync` 拷回 host，CPU 做 heatmap NMS、descriptor 采样、互反查匹配、y/disp/MAD/final gate。
- 状态: 已通过 NX TensorRT extractor 管线验证；128/160/224 ROI engine 均可 `trtexec`。最新实时矩阵使用 XFeat 128/top64，`Stage2_NeuralFeatureMatch` 平均 `4.28ms`，async worker 平均 `4.41ms`，约 `90.3fps`，但只有 `20/583` 帧生成有效神经深度；relaxed 诊断为 `0/575`，不是默认深度源。

通用 direct extractor fallback 也已接入:

- 条件: 无 fused engine, 且 extractor engine 输出可识别的固定 shape `keypoints`、`descriptors`、可选 `scores`。
- 支持 schema: keypoints 为 `[N,2]` 或 `[N,3]`, descriptors 为 `[N,D]` 或 `[D,N]`, `D` 等于 `descriptor_dim`。
- 后处理: TensorRT extractor 在 GPU 上跑；输出回 host 后做真实 descriptor mutual-NN、positive disparity、y/disp/MAD/final gate。
- 用途: 给 ALIKED、SuperPoint extractor-only 或其他导出为点+描述子的真实模型提供实时接线入口；没有匹配点输出时不会生成模拟候选。2026-07-03 clean rerun 中 SuperPoint 固定 schema engine 可以运行，但实时矩阵只有 `5/114` 帧有效，async worker 平均 `14.60ms`、over-deadline `332`；ALIKED 未生成 engine。

固定 schema split matcher 路径也已接入:

- 条件: 无 fused engine, `use_lightglue=true`, 同时配置 `extractor_engine_path` 和 `matcher_engine_path`。
- 输入: extractor 先在左右 ROI 各跑一次；C=1 使用 gray，C=3 必须使用 BGR 管线。C++ 将 extractor 解析出的 keypoints/descriptors/可选 scores 填入 matcher 输入。
- 支持 matcher 输入: 左右 `keypoints/kpts/coords`, `descriptors/desc`, 可选 `scores/conf/prob`, 可选 `image_size`；名称中 `left/right/query/train` 或 `0/1` 语义后缀用于分左右，无法判断时按同类型输入顺序分配。
- 支持 matcher 输出: `matches [M,2]`、`matches [M,3]`，或左到右 `matches0` 加可选 `scores0/mscores0`。`matches1` 右到左索引不会被当成左到右结果使用。
- 后处理: matcher 输出 D2H 后仍执行 positive disparity、y/disp/MAD/final gate。它是真 TensorRT matcher 推理，但不是完整 GPU 常驻深度验证。

限制:

- split matcher 只覆盖上述固定 schema；需要 mask、length、动态 top-k 或其他额外状态的 LightGlue/匹配器 engine 仍可能 unsupported。推荐优先导出 fused `[N,4/5]` 点对输出 engine。
- direct extractor fallback 仍有 D2H 拷贝和 CPU mutual-NN 后处理，不能当作完整 GPU 常驻匹配路径。
- 彩色神经 engine 需要检测/取流侧启用 BGR 管线；否则实时 matcher 会返回 `unsupported_input_schema`，不会把灰度指针当 3 通道读。
- 没有 TensorRT engine 时不能把 Python torch probe 的结果直接等价为 NX 实时性能。
- 神经特征默认关闭，必须先用基准片段和 NX 矩阵测试证明耗时和稳定性。

## 错误匹配剔除

实时和离线都应保持一致的验证顺序:

1. 点落在左右投影 overlap mask 内。
2. 校正后 y 残差小于配置门限。
3. 左右反查一致。
4. 正视差且在最大视差范围内。
5. disparity 走 weighted median 或 RANSAC/MAD，避免简单平均。
6. 三角化点接近球体物理半径。
7. 与 bbox/circle 中心深度不应出现无法解释的大偏差。

若存在稳定 y 倾斜，优先检查标定；确实是系统残差时再用 `feature_y_slope` 和 `feature_y_offset_px` 显式建模。

## 连续帧验收

新特征方法进入实时默认配置前，必须在同一 baseline clip 上给出:

- 每帧有效点数。
- validation pass rate。
- 深度中位数和 MAD。
- 帧间深度抖动。
- 每帧耗时。
- 球运动模型残差。
- 典型通过/失败帧的左右匹配图。

只看单张 contact sheet 不能作为启用依据。
