# 性能测试

## 记录格式

任何性能结论必须包含:

- 日期。
- git commit。
- NX 型号和 JetPack 版本。
- 配置文件路径。
- 模型 engine 路径和精度。
- 运行命令。
- 输入数据来源。
- 是否启用显示、记录、baseline clip、debug 输出。
- 平均值、P95/P99 或最大值，而不是只写单个最好值。

## 必测项

| 项目 | 目标 |
|---|---|
| PWM/水印 | 触发频率稳定，左右 frame counter delta 为 0 |
| 取流 | USB 到达抖动不导致长期积压 |
| 检测 | YOLO 推理不阻塞采集 |
| ROI 后处理 | 落后时可丢旧帧 |
| 写盘 | CSV/图片队列限流，不无限增长 |
| GPU/CPU 拷贝 | 不在主路径出现高频无意义往返 |
| 整体 fps | 接近 100Hz 触发频率 |

水印字段解释和独立探针见 [相机同步与水印](相机同步与水印.md)。实时 profile 名称和 Stage 级耗时解释见 [实时调度细节](实时调度细节.md)。

## 推荐命令

系统状态:

```bash
tegrastats --interval 1000
```

TensorRT engine:

```bash
trtexec --loadEngine=/home/nvidia/NX_volleyball/model/yolo26_gpu_fp16.engine
```

全管线:

```bash
./stereo_pipeline --config ../config/pipeline_dual_yolo_roi.yaml
```

基准录制:

```bash
./stereo_pipeline \
  --config ../config/pipeline_dual_yolo_roi.yaml \
  --record-baseline-clip \
  --baseline-duration 3.0 \
  --baseline-image-mode bgr \
  --baseline-format png
```

基准录制默认先缓存到内存再写盘。评估实时影响时重点看 `Stage2_FrameCB`，写盘时间发生在 clip 捕获完成后，不应计入捕获阶段的帧间耗时。

## 回归标准

合入实时路径改动前至少确认:

- 无编译错误。
- 默认配置能启动。
- 100Hz 触发下水印稳定。
- 启用记录不会明显拖慢主循环。
- 打开新增算法时有独立耗时统计。
- 关闭新增算法时性能回到基线。

## GPU 锁频对帧率的影响（2026-07-07 实测）

NX Orin NX 在 MAXN_SUPER 模式下，GPU 默认使用 `nvhost_podgov` 动态调频（306-1173 MHz）。
未执行 `sudo jetson_clocks` 时，GPU 从最低频 306 MHz 起步，YOLO 推理耗时波动导致 stale drop 暴增。

| 配置 | GPU 频率 | 60s 输出帧 | FPS | DropStale | AsyncRoiWorker avg/p95 | Stage0_WaitGrab p95 |
|---|---|---|---|---|---|---|
| 未锁频 | 306 MHz（动态） | 4245 | ~70 | 545 | 5.44 / 7.06 ms | 8.95 ms |
| 锁频 | 1173 MHz（固定） | 5282 | **90.0** | **2** | 4.54 / 6.39 ms | 1.87 ms |

**结论**：`sudo jetson_clocks` 是实时管线运行的硬前提。测试脚本 `nx_p1_dy_regression.sh` 已加入自动检测和锁频。
