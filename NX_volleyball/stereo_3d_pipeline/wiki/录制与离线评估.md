# 录制与离线评估

本页说明 baseline clip 图像片段录制和离线图像算法评估。轨迹模型训练主数据不是 baseline clip，而是实时 `TrajectoryRecorder` CSV；采集步骤见 [轨迹模型数据采集流程](轨迹模型数据采集流程.md)。

## 基准数据格式

对特征点和深度算法验证，首选“左右图片序列 + CSV”，而不是只录视频。

原因:

- 图片序列保留逐帧文件名和随机访问能力。
- PNG/PGM 是无损格式，不引入视频编码伪影。
- CSV 可以记录检测框、水印、同步差、触发索引和运行状态。
- 后续算法可以在同一帧集上重复比较，避免每次重新拍摄造成输入差异。

视频可以作为人工预览或演示输出，但不作为算法评估主数据。

## 实时录制行为

baseline clip 录制接在 `stereo_pipeline` 的实时帧回调里，输入是校正后的左右灰度图、可选左右 BGR 图、左右 YOLO 检测框和相机水印。进入录制模式后，程序会切到 detection-only:

- 仍运行左右 YOLO，保证 `frames.csv` 有检测框。
- 关闭深度融合、tracker、神经特征和双目深度输出，降低录制时的额外负载。
- 不启动轨迹 CSV `TrajectoryRecorder`; 当前模式只生成图像序列、`frames.csv` 和 `metadata.yaml`。
- 默认 `write_after_capture=true`，实时回调只把图像复制到内存队列，clip 完成后由后台线程写 PNG/PGM。

彩色录制依赖 BGR 管线。使用 `--baseline-image-mode bgr` 或 `both` 时，配置中的 `detector.input_format` 应为 `bgr`; 否则程序会告警，输出不应作为彩色算法评估基准。

## 录制命令

```bash
cd ~/NX_volleyball/stereo_3d_pipeline/build_standalone
./stereo_pipeline \
  --config ../config/pipeline_dual_yolo_roi.yaml \
  --record-baseline-clip \
  --baseline-out baseline_clips \
  --baseline-duration 3.0 \
  --baseline-clips 3 \
  --baseline-gap 1.0 \
  --baseline-image-mode bgr \
  --baseline-format png
```

`--baseline-frames <n>` 可以替代 duration，精确控制每段 clip 的帧数。`--baseline-start-immediately` 会跳过检测触发条件，适合只采连续左右目图像。

录制器默认先把图像快速缓存到内存，clip 完成后再写盘，避免 PNG 写盘阻塞实时管线。`--baseline-image-mode gray` 保存灰度图，`--baseline-image-mode bgr` 保存彩色图；`both` 只用于调试兼容。

如果命令行报 `unknown option(s): --record-baseline-clip ...`，不是命令语法问题，而是当前 `build_standalone/stereo_pipeline` 没有编进 baseline CLI。先用 `./stereo_pipeline --help | grep baseline` 和 `strings ./stereo_pipeline | grep record-baseline` 检查二进制；再用 `grep -n "record-baseline" ../src/main.cpp` 检查源码。源码有参数但二进制没有时，需要重新构建 `stereo_pipeline`。

也可以直接在 YAML 里启用:

```yaml
baseline_recording:
  enabled: true
  output_dir: "baseline_clips"
  frame_limit: 200
  image_mode: "bgr"
  image_format: "png"
  write_after_capture: true
```

命令行参数会覆盖 YAML 中的同名录制字段。临时采集建议用命令行覆盖，长期固定采集流程再写进配置。

常用模式:

| 场景 | 推荐参数 |
|---|---|
| 离线特征点/彩色 patch 测试 | `--baseline-image-mode bgr --baseline-format png` |
| 灰度关键点或快速大批量采集 | `--baseline-image-mode gray --baseline-format pgm` |
| 不等检测框, 直接录连续图 | `--baseline-start-immediately` |
| 多段采样 | `--baseline-clips N --baseline-gap SEC` |

内存策略:

- `write_after_capture=true` 时，实时回调只做 VPI/CUDA 图像到 `cv::Mat` 的内存复制和入队。
- 写 PNG/PGM 会等全部 clip 捕获完成后再执行。
- `max_queue_frames=0` 表示不限队列；长时间彩色录制前要确认 NX 内存余量。
- 如果设置正数上限，队列满时丢录制帧，不反向阻塞 100fps 管线。
- `write_after_capture=true` 且队列上限小于目标帧数时，代码会把有效上限抬到 `target_frames * clip_count`，避免还没写盘就提前丢帧。

## 输出内容

```text
clip_YYYYMMDD_HHMMSS_01/
├── left/
├── right/
├── left_bgr/      # image_mode=both 时存在
├── right_bgr/     # image_mode=both 时存在
├── frames.csv
└── metadata.yaml
```

`frames.csv` 记录:

- pipeline frame id。
- 左右图片相对路径。
- 额外 BGR 图片相对路径；仅 `image_mode=both` 时填写，`bgr` 模式下 `left_image`/`right_image` 本身就是彩色图。
- 左右检测数量和最佳检测框。
- 左右配对是否通过。
- disparity、dy、size ratio。
- 左右相机 timestamp、frame number、frame counter、trigger index。
- frame counter delta、timestamp delta。
- `grab_failed` 和 `is_detect_frame` 运行状态。

录制后检查:

```bash
CLIP=baseline_clips/clip_YYYYMMDD_HHMMSS_01
find "$CLIP/left" -type f | wc -l
find "$CLIP/right" -type f | wc -l
wc -l "$CLIP/frames.csv"
sed -n '1,40p' "$CLIP/metadata.yaml"
```

`frames.csv` 行数应为图片帧数加 1 行表头；`metadata.yaml` 中应记录 `image_mode`、`write_after_capture`、`target_frames` 和 `trigger_hz`。

关键 CSV 字段:

| 字段 | 含义 |
|---|---|
| `clip_frame_id`, `pipeline_frame_id` | clip 内序号和实时管线帧号 |
| `left_image`, `right_image` | 主图像相对路径；由 `image_mode` 决定灰度或彩色 |
| `left_bgr_image`, `right_bgr_image` | 仅 `image_mode=both` 时填写 |
| `left_count`, `right_count` | 左右 YOLO 检测数量 |
| `best_left_idx`, `best_right_idx` | 参与记录的最佳检测框索引 |
| `pair_valid`, `pair_disparity_px`, `pair_dy_px`, `pair_size_ratio` | 简单左右框配对诊断 |
| `left_*`, `right_*` bbox 字段 | 最佳左右检测框中心、尺寸、置信度和类别 |
| `left_timestamp_us`, `right_timestamp_us` | 相机水印时间戳 |
| `left_frame_counter`, `right_frame_counter` | 相机帧计数 |
| `left_trigger_index`, `right_trigger_index` | 触发序号 |
| `frame_counter_delta`, `frame_number_delta`, `timestamp_delta_us` | 左右同步诊断 |
| `grab_failed`, `is_detect_frame` | 抓帧和检测帧状态 |

baseline clip 的 `frames.csv` 只记录触发和简单检测诊断。实时轨迹 CSV 需要在配置中启用 `recording.detail_level: depth_candidates` 或 `extended`，才能记录 `pair_initial_disparity`, `pair_shifted_iou`, `pair_score`, `pair_bbox_prior_penalty`, `pair_positive_disparity` 以及各深度候选的 `support/std/confidence`。

实时 recorder 还会写 `*.frames.csv` sidecar，对每个进入实时 recorder 的结果回调帧记录同步水印、`result_count`、`direct_pair_count`、`fallback_l2r_count`、`fallback_r2l_count`、selected pair 的 IoU/score/penalty、关键特征 support，以及 P2 `p2_depth_mode_mask`、realtime/diagnostic requested 和 trigger mask。这个文件用于统计已发布结果帧内的无输出、连续运行误匹配退化和 P2 是否被触发，不能用 baseline clip 的 `frames.csv` 替代；它仍不包含被 async deadline 丢弃且未发布的 stale 帧。

## 单帧可视化

```bash
./stereo_pipeline \
  --config ../config/pipeline_dual_yolo_roi.yaml \
  --debug-feature-matches \
  --debug-feature-matches-dir debug_feature_matches
```

用于快速查看左右检测框、legacy CPU/离线 ROI 匹配点和错误匹配形态。它不是连续稳定性评估，也不输出 P2 GPU/VPI/TRT 后端内部匹配，不能替代基准片段测试。

## YOLO/IoU 退化回归

灰度连续 clip 可以先验证左右 YOLO 配对、假框注入和单侧漏检 fallback 是否稳定:

```bash
python3 NX_volleyball/stereo_3d_pipeline/tools/offline_yolo_iou_fallback_regression.py \
  --clip NX_volleyball/stereo_3d_pipeline/test_logs/nx_recordings/codex_2s_gray_20260702_142253/clip_20260702_142255_01 \
  --calib NX_volleyball/calibration/stereo_calib.yaml \
  --out NX_volleyball/stereo_3d_pipeline/test_logs/yolo_iou_fallback_regression_latest \
  --max-frames 0 \
  --fail-on-regression
```

这个测试不验证彩色 LAB/label patch 特征质量。彩色 IoU/patch 和神经模型输入转换必须使用 BGR PNG clip 单独评估。

## 离线评估流程

1. 录制基准片段。
2. 固定相同输入，逐个启用特征提取和匹配方法。
3. 自动计算有效点数、pass rate、深度中位数、MAD、帧间抖动、耗时和运动模型残差。
4. 只把通过连续帧验证的方法纳入实时管线候选。
5. 生成图片 contact sheet 给人检查，但决策以脚本指标为准。

彩色 PNG 连续帧评估:

```bash
python3 NX_volleyball/stereo_3d_pipeline/tools/offline_volleyball_sequence_probe.py \
  --clip baseline_clips/clip_YYYYMMDD_HHMMSS_01 \
  --calib NX_volleyball/calibration/stereo_calib.yaml \
  --out NX_volleyball/stereo_3d_pipeline/test_logs/sequence_probe_latest
```

`offline_volleyball_sequence_probe.py` 会逐帧调用单帧 probe，输出 `per_frame_methods.csv`、`method_summary.csv` 和 `report.md`。灰度输入会被 OpenCV 转成 BGR 容器，但没有真实颜色信息；彩色方法的结论只认 BGR PNG 录制。
