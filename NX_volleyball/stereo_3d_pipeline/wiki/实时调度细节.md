# 实时调度细节

最后核对: 2026-07-02

实时调度核心在 `src/pipeline/pipeline.cpp` 的 `pipelineLoopROI()`。本页解释当前 100fps ROI 路径的帧生命周期、丢帧策略和同步点。

## 总体流水

ROI 模式采用三段重叠:

```text
requestGrab(N+1)
  -> Stage1 Detect(N)
  -> Stage2 Fuse(N-1)
  -> waitGrab(N+1)
  -> Stage0 Process(N+1)
```

目标是让 USB 取流等待、YOLO enqueue、ROI 后处理和校正提交尽量重叠。100Hz 触发下，每帧预算约 10ms，实时性依赖异步采集、CUDA event、VPI stream 和写盘队列都不阻塞主路径。

## Stage0: 采集和校正

相关函数: `stage0_grab_and_rectify()`

当前行为:

- 相机模式下先由采集线程写入 `FrameSlot.rawL/rawR`。
- 记录左右 timestamp、frame number、frame counter、trigger index。
- 新一帧 remap 前同步 VPI stream，避免覆盖仍在使用的 buffer。
- 可选 VPI TNR: U8 -> NV12 -> TNR -> U8。
- BGR 检测路径:
  - Bayer -> BGR8 CUDA kernel。
  - VPI/CUDA remap 输出校正 BGR。
  - BGR -> Gray 供 stereo/ROI 使用。
- 灰度 legacy 路径直接输出校正 gray。
- 函数提交 remap 后不做最终同步，Stage1 在检测前同步。

## Stage1: 检测提交

相关函数: `stage1_detect()`

当前行为:

- 检测帧按 `effective_detect_interval_` 固定节拍提交 YOLO。
- 左目使用主 TensorRT detector。
- `dual_yolo.enabled=true` 时右目也提交 detector。
- `stage1_detect()` 只 enqueue，不在提交路径等待推理完成。
- 每个 ring slot 有对应 CUDA event，提交后调用 `recordDetectDoneEvents()` 记录完成事件。

Tracker 填充帧:

- 当 tracker 启用且当前帧不是检测帧，等待校正完成后执行 tracker infill。
- 当前代码为了控制 NX 功耗，会在 tracker infill 前显式等待已提交 YOLO 完成。
- tracker 结果会通过兼容 event 让 Stage2 使用同一等待逻辑。

tracker 的配置、状态和补帧风险见 [追踪与补帧](追踪与补帧.md)。

## Stage2: ROI 匹配、融合和回调

相关函数:

- `stage2_roi_match_fuse()`
- `stage2_roi_fuse_tracker()`
- `waitDetectDone()`
- `detectEventsReady()`

检测帧:

1. 等待当前 slot 的 detect event。
2. 同步 DLA/GPU stream，确保 YOLO 输出可 collect。
3. 收集左右 YOLO 检测。
4. 执行 dual YOLO 配对、GPU ROI 候选、可选 CPU/神经特征候选。
5. 进入 hybrid depth 和轨迹预测。
6. 刷新 tracker template。

填充帧:

1. 使用 tracker bbox。
2. 执行 ROI depth/fusion。
3. 不刷新 YOLO 检测结果。

Stage2 结束后调用:

- `result_callback_`，用于结果消费。
- `frame_callback_`，用于显示、基准片段录制和外部观察。

## 旧 ROI 后处理丢弃策略

配置字段:

```yaml
performance:
  drop_stale_roi_frames: true
```

当前丢弃发生在 Stage2 进入 ROI 后处理前。条件是:

- 当前帧是检测帧。
- 后面至少已有一帧进入检测队列。
- 下一帧对应 slot 的 YOLO event 已 ready。
- 当前不是 `detection_only` 模式。

因此它丢的是“落后的 ROI/深度后处理”，不是 USB 取流帧本身，也不是 YOLO 刚提交就立刻丢弃。这样可以在后处理耗时过高时降低端到端延迟，但仍需要通过水印确认采集本身稳定。

## GPU 友好边界

当前热路径的 GPU 友好点:

- 校正后灰度 CUDA 指针缓存到 `FrameSlot`，ROI GPU matcher 直接使用，避免每帧 VPI lock/unlock。
- 双目 ROI 几何、patch、sparse-lite 候选可在一个 CUDA batch 中运行。
- TensorRT detector 使用 slot 级 context/buffer，减少资源竞争。
- 记录和图片写盘走异步队列。

当前不是严格端到端零拷贝:

- 海康 SDK buffer 会拷贝到调用方 VPI host-mapped buffer。
- YOLO collect、CSV 记录和 debug 图片会回到 CPU。
- baseline clip 需要复制校正灰度图到 `cv::Mat` 后由后台线程写盘。
- XFeat extractor-only TensorRT split 已可运行，但当前会把 `feats/keypoints/heatmap` 拷回 CPU 做关键点选择和匹配。
- ALIKED、SuperPoint+LightGlue 的真实 TensorRT split runtime 尚未完整实现；fused `[N,4/5]` 输出入口已支持。
- true OpenCV CUDA ORB 已接入 `roi_orb_points`，但当前同步路径会阻塞 Stage2，实测不守 100fps。

新增算法时必须标注它是否在 GPU 上完成、是否需要 CPU round-trip、是否引入 stream sync。

## 性能观测点

当前 profile 名称包括:

| 名称 | 含义 |
|---|---|
| `Stage0_WaitGrab` | 等待异步相机取流完成 |
| `Stage0_Process` | Bayer、TNR、校正和灰度/BGR 处理提交 |
| `Stage1_WaitRect` | 检测前等待校正完成 |
| `Stage1_DetectSubmit` | YOLO enqueue 时间 |
| `Stage1_WaitYOLOComplete` | tracker 填充路径等待 YOLO 完成 |
| `Stage2_WaitDetect` | 等待 detect event |
| `Stage2_WaitYOLOComplete` | collect 前等待 YOLO stream |
| `Stage2_DualYoloMatch` | 左右检测配对和候选处理 |
| `Stage2_DualYoloGpuCandidates` | GPU ROI 候选耗时 |
| `Stage2_OpenCVCudaORB` | true OpenCV CUDA ORB ROI 提取和匹配耗时 |
| `Stage2_NeuralFeatureMatch` | TensorRT 神经特征推理和当前后处理耗时 |
| `Stage2_ROIMatchFuse` | ROI 候选和融合总耗时 |
| `Stage2_FrameCB` | 显示/录制等帧回调耗时 |
| `Stage2_DropStaleROI` | 丢弃旧 ROI 后处理计数 |

实机 review 时至少检查 FPS、`stale_drop`、水印 delta、`Stage0_WaitGrab` outlier 和 Stage2 GPU 候选耗时。
