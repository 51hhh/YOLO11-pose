# 实时调度细节

最后核对: 2026-07-03

实时调度核心在 `src/pipeline/pipeline_loops.cpp` 的 `pipelineLoopROI()`。异步 ROI Stage2 的 snapshot、worker、完成队列和过期丢弃在 `src/pipeline/pipeline_async_roi.cpp`；Stage0/1/2/3 主体、dual YOLO ROI 候选和融合逻辑仍在 `src/pipeline/pipeline.cpp`。本页解释当前 100fps ROI 路径的帧生命周期、丢帧策略和同步点。

## 总体流水

ROI 模式采用三段重叠:

```text
requestGrab(N+1)
  -> Stage1 Detect(N)
  -> Stage2 Fuse(N-1)
  -> waitGrab(N+1)
  -> Stage0 Process(N+1)
```

目标是让 USB 取流等待、YOLO enqueue、ROI 后处理和校正提交尽量重叠。100Hz 触发下，每帧预算约 10ms，实时性依赖异步采集、CUDA event、VPI stream 和写盘队列都不阻塞主路径。

## Stage0: 采集和校正

相关函数: `stage0_grab_and_rectify()`

当前行为:

- 相机模式下先由采集线程写入 `FrameSlot.rawL/rawR`。
- 记录左右 timestamp、frame number、frame counter、trigger index。
- 新一帧 remap 前同步 VPI stream，避免覆盖仍在使用的 buffer。
- 可选 VPI TNR: U8 -> NV12 -> TNR -> U8。
- BGR 检测路径:
  - Bayer -> BGR8 CUDA kernel。
  - VPI/CUDA remap 输出校正 BGR。
  - BGR -> Gray 供 stereo/ROI 使用。
- 灰度 legacy 路径直接输出校正 gray。
- 函数提交 remap 后不做最终同步，Stage1 在检测前同步。

## Stage1: 检测提交

相关函数: `stage1_detect()`

当前行为:

- 检测帧按 `effective_detect_interval_` 固定节拍提交 YOLO。
- 左目使用主 TensorRT detector。
- `dual_yolo.enabled=true` 时右目也提交 detector。
- `stage1_detect()` 只 enqueue，不在提交路径等待推理完成。
- 每个 ring slot 有对应 CUDA event，提交后调用 `recordDetectDoneEvents()` 记录完成事件。

Tracker 填充帧:

- 当 tracker 启用且当前帧不是检测帧，等待校正完成后执行 tracker infill。
- 当前代码为了控制 NX 功耗，会在 tracker infill 前显式等待已提交 YOLO 完成。
- tracker 结果会通过兼容 event 让 Stage2 使用同一等待逻辑。

tracker 的配置、状态和补帧风险见 [追踪与补帧](追踪与补帧.md)。

## Stage2: ROI 匹配、融合和回调

相关函数:

- `stage2_roi_match_fuse()`
- `stage2_roi_fuse_tracker()`
- `waitDetectDone()`
- `detectEventsReady()`

检测帧同步路径:

1. 等待当前 slot 的 detect event。
2. 同步 DLA/GPU stream，确保 YOLO 输出可 collect。
3. 收集左右 YOLO 检测。
4. 执行 dual YOLO 配对、GPU ROI 候选、可选 CPU/神经特征候选。
5. 进入 hybrid depth 和轨迹预测。
6. 刷新 tracker template。

填充帧:

1. 使用 tracker bbox。
2. 执行 ROI depth/fusion。
3. 不刷新 YOLO 检测结果。

Stage2 结束后调用:

- `result_callback_`，用于结果消费。
- `frame_callback_`，用于显示、基准片段录制和外部观察。

检测帧异步路径:

当 `performance.async_roi_stage2=true` 且 ROI_ONLY、非 detection_only、tracker 未启用或 `detect_interval<=1` 时，检测帧不再在 pipeline 主线程内执行完整 `stage2_roi_match_fuse()`。当前主配置 `tracker.enabled=false`，所以异步 ROI Stage2 会生效；如果启用 tracker 且 `detect_interval>1`，代码会自动退回同步 Stage2，避免异步检测帧结果倒序更新 tracker/深度滤波状态。主线程在 YOLO ready 后只做:

1. collect 左右 YOLO 检测。
2. 将当前 slot 的校正灰度图复制到 async ROI worker 自有 GPU buffer。
3. 当帧左右目都有检测，且启用彩色候选或神经 TensorRT engine 的输入 schema 需要 3/6 通道时，复制校正 BGR 图到 worker 自有 GPU buffer。
4. 当帧确实可能进入 CPU fallback/真实 OpenCV CPU 特征时，复制灰度图到 pinned host buffer。
5. 提交 `AsyncRoiTask` 后继续后续调度。

如果左右检测都为空，async 提交会直接早退为 predict-only，不申请 async buffer，也不做 gray/BGR/host snapshot。只有右目单检 fallback 可能运行时，才会在右目非空、左目为空的帧继续提交 Stage2。

worker 线程随后在低优先级 CUDA stream 上执行完整 `runRoiStage2Core()`，包括 IoU/极线配对、ROI 候选、OpenCV CUDA ORB、CPU OpenCV BRISK/AKAZE/SIFT、TensorRT XFeat/ALIKED/SuperPoint/LightGlue、颜色候选、亚像素候选和候选融合。worker 不直接调用 callback，也不直接更新外部输出；完成后只把 `RoiStage2Output` 放入完成队列。pipeline 主线程通过 `drainCompletedAsyncRoiStage2()` 接收未过期结果，再执行 `applyRoiStage2Output()` 和 callback。

当前锁边界:

- `roi_postprocess_mutex_` 只保护共享 ROI matcher、OpenCV/TensorRT feature matcher 和相关临时上下文，避免多个 Stage2 同时复用内部 buffer。
- `applyRoiStage2Output()` 不持有 `roi_postprocess_mutex_`；它只在主线程写回 slot，并用 `hybrid_depth_mutex_` 保护 `HybridDepthEstimator` 和 `last_fuse_time_`。
- async 提交失败时的 predict-only 回退不会等待正在运行的重特征 worker，因此不会因为 fallback 输出再把主调度卡住。

## 算法输入和内存调度

异步 Stage2 只在提交任务时从 `FrameSlot` 复制算法实际需要的数据。当前策略:

| 路径 | 实时输入 | 计算位置 | async buffer 策略 |
|---|---|---|---|
| 几何、center patch、multi-point、corner/texture/binary sparse-lite | 校正 gray GPU | 自研 CUDA kernel | 只需要 `left/right_gray_gpu` D2D 快照 |
| 彩色 IoU/edge patch | 校正 BGR GPU | 自研 CUDA kernel | 仅在彩色候选启用且 BGR 管线可用时分配/复制 `left/right_bgr_gpu` |
| OpenCV CUDA ORB | 校正 gray GPU | OpenCV CUDA ORB + CUDA matcher | 不需要 host 图；keypoints/matches 元数据回 CPU gate；不为 CPU ORB fallback 单独复制 host gray |
| OpenCV BRISK/AKAZE/SIFT | 校正 gray host | CPU OpenCV | 仅这些开关或 CPU fallback 可能运行时分配/复制 pinned host gray |
| XFeat extractor-only | 1ch gray 或 3ch BGR GPU ROI | TensorRT extractor + CPU 后处理 | 输入不需要 host 图；模型输出 D2H 后做 NMS/descriptor/mutual-NN |
| ALIKED/SuperPoint direct extractor | 1ch gray 或 3ch BGR GPU ROI | TensorRT extractor + CPU descriptor matching | 输入不需要 host 图；要求真实 engine 输出 `keypoints/descriptors/scores` |
| direct extractor split matcher | 1ch gray 或 3ch BGR GPU ROI | TensorRT extractor + TensorRT matcher + CPU gate | 输入不需要 host 图；仅 `use_lightglue=true` 时启用；只支持固定 keypoint/descriptor 输入和 `matches`/`matches0` 输出 schema |
| fused neural matcher | 2ch gray-pair 或 6ch BGR-pair GPU ROI，也支持两个 1/3ch 输入 | TensorRT fused matcher + CPU gate | 输入不需要 host 图；输出点对 D2H 后做几何 gate |
| epipolar/template/feature fallback | 校正 gray host | CPU fallback | 只有对应 fallback 可能触发时才分配 pinned host gray |

`AsyncRoiBuffer.left/right_gray_host` 不是 GPU-only 配置的默认分配项。初始化时，只有 OpenCV CPU BRISK/AKAZE/SIFT 或 CPU fallback 在配置上可能运行，才会预分配 pinned host gray；提交任务时再按当帧检测结果决定是否 D2H。`roi_orb_points` 在 `gpu_candidate_refine=true` 下使用 CUDA ORB，不会单独触发 host gray 分配或 D2H；CPU ORB fallback 只会在关闭 GPU refine，或其他 CPU 路径已经提供 host gray 时发生。

每帧 snapshot 收紧规则:

- 左右检测都为空: 不申请 async buffer，不复制 gray/BGR/host，直接 predict-only。
- 单侧检测、多检测框歧义，或一左一右无法通过预配对 gate: CPU fallback 可能运行，才复制 pinned host gray。
- 一左一右且可预配对: GPU 几何/patch/ORB/神经路径只复制 gray GPU；只有启用 BRISK/AKAZE/SIFT CPU 描述子时才复制 host gray。
- 彩色候选或 3/6 通道神经 engine: 只有 BGR 管线启用且当帧有左右配对机会时复制 BGR GPU snapshot；gray-only 管线返回 unsupported input，不会误读 gray 指针。

## ROI 后处理丢弃策略

配置字段:

```yaml
performance:
  drop_stale_roi_frames: true
  async_roi_stage2: true
  async_roi_buffers: 3
  async_roi_deadline_ms: 10.0
```

同步旧策略仍保留: 丢弃发生在 Stage2 进入 ROI 后处理前。条件是:

- 当前帧是检测帧。
- 后面至少已有一帧进入检测队列。
- 下一帧对应 slot 的 YOLO event 已 ready。
- 当前不是 `detection_only` 模式。

异步策略新增两层丢弃:

- 当当前检测帧 YOLO 已 ready 时，调用 `expireAsyncRoiBefore(frame_id)`，把更老的 pending/ready async ROI 结果标记过期。
- 如果 worker 正在处理旧帧，无法强杀已经运行的 CUDA/CPU work；它完成后会释放 buffer，但结果因 `frame_id < expire_before` 被丢弃。
- 如果 async buffer 不足，会替换最老 pending task；running task 不被抢占。
- 如果提交失败，当前帧回退为 predict-only 输出，避免阻塞主调度。

因此它丢的是“过期的 ROI/深度后处理结果”，不是 USB 取流帧本身，也不是已经提交的 YOLO 推理。这样可以给 ORB/XFeat/亚像素等重候选最多约一帧时间窗口；下一帧 YOLO ready 后仍未完成的结果不会再写回主输出。

## 异步内存生命周期

`FrameSlot` 中的 `rectGray_*_gpu.data` 和 `rectBGR_*_gpu.data` 仍然只属于 ring buffer。异步 Stage2 不能长期持有这些指针，因此当前实现使用 async worker 自有缓冲:

- `AsyncRoiBuffer.left/right_gray_gpu`: worker 自有校正灰度快照。
- `AsyncRoiBuffer.left/right_bgr_gpu`: 彩色候选或 3/6 通道神经 engine 需要时分配和复制。
- `AsyncRoiBuffer.left/right_gray_host`: CPU fallback 或 OpenCV CPU BRISK/AKAZE/SIFT 可能运行时才分配，使用 pinned host copy。
- 每个 async buffer 有 `copy_done` event；worker 开始 Stage2 前等待这个 event。
- 每个 ring slot 有 `async_roi_slot_copy_done_` event；slot 下次 `reset()` 前只等待 snapshot copy 完成，不等待 worker Stage2 完成。
- pending task 被替换、过期或 shutdown 回收时，先从队列摘除 buffer，再在 `async_roi_mutex_` 外等待 copy event，最后把 buffer 放回 free list；避免持队列锁等待 CUDA copy。

生命周期如下:

```text
YOLO ready
  -> collect detections
  -> FrameSlot 图像指针只用于 async snapshot enqueue
  -> 记录 slot copy event
  -> 提交 AsyncRoiTask
  -> 主线程继续调度下一帧

worker:
  -> 等 buffer copy_done
  -> 在 worker 自有 GPU/host buffer 上执行完整 Stage2
  -> 完成后入 completed queue

下一次复用同一个 ring slot:
  -> 等 slot copy event
  -> reset FrameSlot
  -> 新图像可覆盖原 slot 内容
```

这个边界能处理“第一帧 worker 仍在跑、第三帧 ring slot 即将复用”的情况: 第三帧只需要等第一帧 snapshot copy 完成，而不需要等第一帧 ORB/XFeat/ROI Stage2 完成。worker 后续读取的是自有快照缓冲。

## GPU 调度边界

异步 ROI worker 使用低优先级 CUDA stream。这样当 YOLO 使用 GPU engine 而不是 DLA 时，ROI 后处理会尽量让路给检测推理。需要注意:

- CUDA stream priority 是调度 hint，不能抢占已经开始运行的 kernel。
- running task 不能强杀，只能完成后丢弃结果。
- `roi_postprocess_mutex_` 串行保护共享 matcher/context，防止 TensorRT/OpenCV matcher 内部 buffer 被并发复用；它不包住结果写回。
- `hybrid_depth_`、`last_fuse_time_` 和 callback 仍只在主线程应用 accepted result，避免 worker 乱序更新轨迹状态；相关滤波状态由 `hybrid_depth_mutex_` 单独保护。

## GPU 友好边界

当前热路径的 GPU 友好点:

- 校正后灰度 CUDA 指针缓存到 `FrameSlot`，ROI GPU matcher 直接使用，避免每帧 VPI lock/unlock。
- 双目 ROI 几何、patch、sparse-lite 候选可在一个 CUDA batch 中运行。
- TensorRT detector 使用 slot 级 context/buffer，减少资源竞争。
- 记录和图片写盘走异步队列。

当前不是严格端到端零拷贝:

- 海康 SDK buffer 会拷贝到调用方 VPI host-mapped buffer。
- YOLO collect、CSV 记录和 debug 图片会回到 CPU。
- baseline clip 按 `image_mode` 复制校正灰度或 BGR 到内存队列，默认 clip 结束后再由后台线程写盘。
- XFeat extractor-only TensorRT split 已可运行，但当前会把 `feats/keypoints/heatmap` 拷回 CPU 做关键点选择和匹配。
- ALIKED/SuperPoint direct extractor schema 已可接线到 descriptor mutual-NN fallback；`matcher_engine_path` 在 `use_lightglue=true` 时支持固定 schema split matcher，但中间特征仍要 D2H/H2D，最终 gate 仍在 CPU；fused `[N,4/5]` 输出入口仍是优先验证路径。
- true OpenCV CUDA ORB 已接入 `roi_orb_points`；旧同步路径或未启用 `async_roi_stage2` 时会阻塞 Stage2，实测不守 100fps。

新增算法时必须标注它是否在 GPU 上完成、是否需要 CPU round-trip、是否引入 stream sync。
