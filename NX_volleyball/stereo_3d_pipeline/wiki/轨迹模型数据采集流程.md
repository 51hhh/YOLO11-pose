# 轨迹模型数据采集流程

最后核对: 2026-07-07

轨迹模型训练使用固定配置 `config/pipeline_record_p0p1.yaml`。当前正式采集口径冻结 P0/P1 + 自研 CUDA NCC + XFeat: 主 CSV 记录 `z_roi_multi_point`、`z_roi_center_patch`、`z_roi_cuda_template_match`、`z_roi_neural_xfeat`，SuperPoint/ALIKED 在该配置中关闭。录制开启 `recording.detail_level: extended`、`raw_mode: true` 和 frame sidecar；diagnostic sidecar 默认关闭，只在临时 debug 配置中打开。训练读取原始候选字段，例如 `z_bbox_center`、`z_circle_center`、`z_roi_edge_centroid`、`z_roi_radial_center`、`z_roi_edge_pair_center`、`z_roi_multi_point`、`z_roi_center_patch`、`z_roi_cuda_template_match`、`z_roi_neural_xfeat`。不要把 `z_stereo` 或 `z` 当训练候选或标签。

## NX 启动

配置或代码更新后先构建一次:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
cmake --build build_standalone -j$(nproc)
```

每段数据使用递增文件名前缀，例如 `traj_p0p1_001`、`traj_p0p1_002`。录制前先写同名前缀的 metadata:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
mkdir -p ~/trajectory_dataset
nano ~/trajectory_dataset/traj_p0p1_001.metadata.yaml
```

metadata 示例:

```yaml
dataset_type: static
scene: center_3m
known_z: 3.0
static: true
notes: static ball, center
```

启动录制:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
sudo jetson_clocks
cd build_standalone
./stereo_pipeline \
  --config ../config/pipeline_record_p0p1.yaml \
  --recording-out ~/trajectory_dataset/traj_p0p1_001.csv \
  2>&1 | tee ~/trajectory_dataset/traj_p0p1_001.log
```

需要抽样查看正式实时路径的算法 zoom 效果时追加低频 dump 参数。它默认后台写盘，输出 `frame_*.png` 和 `frame_*.json`; 正式长时间采集建议先小段验证再打开:

```bash
./stereo_pipeline \
  --config ../config/pipeline_record_p0p1.yaml \
  --recording-out ~/trajectory_dataset/traj_p0p1_001.csv \
  --debug-realtime-dump \
  --debug-realtime-dump-dir ~/trajectory_dataset/traj_p0p1_001.debug \
  --debug-realtime-dump-stride 100 \
  --debug-realtime-dump-max 30 \
  2>&1 | tee ~/trajectory_dataset/traj_p0p1_001.log
```

按 `Ctrl-C` 停止，程序会 flush CSV。下一段把文件名前缀递增为 `traj_p0p1_002`。每段输出:

```text
traj_p0p1_001.csv
traj_p0p1_001.frames.csv
traj_p0p1_001.metadata.yaml
traj_p0p1_001.log
traj_p0p1_001.p2_diagnostic.csv  # optional, only when diagnostic results are enabled
```

正式 `pipeline_record_p0p1.yaml` 默认不写 `.p2_diagnostic.csv`；`z_roi_cuda_template_match` 和 `z_roi_neural_xfeat` 已进入主 CSV。只有临时打开 `p2_diagnostic_results_enabled` 或 debug dump 时，运行时才会根据 `--recording-out ~/trajectory_dataset/traj_p0p1_001.csv` 自动派生 `.p2_diagnostic.csv`。

## 采集内容

先采静态已知距离，再采动态。静态覆盖 `2m, 3m, 4m, 5m, 6m`，每段 60-120s，并在 metadata 写 `known_z`、`static: true`、场景位置。动态覆盖慢速前后移动、左右/上下移动、正常抛球/击球、遮挡或单侧漏检；没有真实距离时 `known_z: null`，但必须写清楚 `scene` 和 `notes`。

采集完成后只做最小确认: `log` 末尾应有 `TrajectoryRecorder: saved ... frames`，同名前缀下应有主 CSV 和 `.frames.csv`。

## P1 dy 回归

当修改 P0/P1 候选、pair dy prior、soft gate 或字段记录后，先在 NX 上跑短时回归。该脚本会使用正式 `pipeline_record_p0p1.yaml`，输出主 CSV、frames sidecar、P2 diagnostic sidecar、日志和字段统计报告:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
scripts/nx_p1_dy_regression.sh --build --duration 20
```

可选打开低频 zoom/debug:

```bash
scripts/nx_p1_dy_regression.sh --build --duration 20 \
  --debug-dump --debug-stride 100 --debug-max 20
```

验收重点:

- `p0p1_dy_center` 明显非零时，`z_roi_center_patch`、`z_roi_multi_point`、`z_roi_cuda_template_match` 不应因为固定 `dy=0` 大面积无效。
- 报告里的 `Circle source groups` 要单独看。`roi_fit/roi_fit` 以外的组里 `z_circle_center` 可能按设计无效，但 direct YOLO pair + GPU candidate 存在时，`z_roi_center_patch` 和 `z_roi_multi_point` 不应再被 circle seed 硬前置挡成全 0。
- `z_bbox_center`、`z_circle_center`、`z_roi_edge_centroid`、`z_roi_radial_center`、`z_roi_edge_pair_center` 应继续保持 P0 高覆盖。
- `p0p1_*_trust` 和 `p0p1_untrusted_mask` 应与候选有效/无效和 dy/disparity 聚类关系一致。
- `Stage2_AsyncRoiWorker`、`Stage2_DualYoloGpuCandidates`、`Stage2_CudaTemplateNccMatch` 仍需满足 100fps 预算；若字段恢复但 worker 长尾变大，先降 P2 频率或关 sidecar，再排算法本身。

## 本地验收和训练入口

下载后先跑离线数据质量检查。它会自动寻找同名前缀 metadata，例如 `traj_p0p1_001.csv` 对应 `traj_p0p1_001.metadata.yaml`:

```bash
cd NX_volleyball/stereo_3d_pipeline
python3 trajectory_fusion/check_dataset.py \
  ~/mid360_datasets/trajectory/traj_p0p1_001.csv \
  --json-out ~/mid360_datasets/trajectory/traj_p0p1_001.check.json
```

重点看 `rows/duration/fps`、`frame_gaps`、`frame_counter_delta/frame_number_delta`、P0/P1 主 CSV 有效率、median、MAD 和 `known_z_bias`。动态片段还要看 `source_breakdown`、`.frames.csv` 汇总、`depth_jump`，确认 direct pair、fallback L2R/R2L、候选深度跳变，以及 `cuda_template`、`neural_xfeat` 的有效率和耗时没有异常。静态已知距离数据如果 `known_z_bias` 明显偏离，应先复查标定、metadata 和同步状态，不要直接并入训练。

多段数据用 manifest 管理，文件路径相对 manifest 所在目录:

```yaml
clips:
  - csv: traj_p0p1_001.csv
    metadata: traj_p0p1_001.metadata.yaml
    split: train
  - csv: traj_p0p1_002.csv
    metadata: traj_p0p1_002.metadata.yaml
    split: val
```

训练前先做 manifest 级预检:

```bash
cd NX_volleyball/stereo_3d_pipeline
.venv-stereo-neural/bin/python trajectory_fusion/validate_dataset_manifest.py \
  ~/mid360_datasets/trajectory/dataset_manifest.yaml \
  --json-out ~/mid360_datasets/trajectory/dataset_manifest.check.json
```

重点看 `split_counts`、`known_z_counts`、`warning_counts` 和每段 `p0_min_hit_rate`。正式模型选择至少应有 `val` split；静态已知距离片段应覆盖 train/val，否则只能验证抖动趋势，不能用绝对 bias/MAD 选模型。

然后跑候选一致性诊断:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/analyze_candidate_consistency.py \
  ~/mid360_datasets/trajectory/dataset_manifest.yaml \
  --reference auto \
  --json-out ~/mid360_datasets/trajectory/candidate_consistency.json \
  --csv-out ~/mid360_datasets/trajectory/candidate_consistency.csv \
  --pairwise-csv-out ~/mid360_datasets/trajectory/candidate_pairwise.csv
```

`--reference auto` 有 `known_z` 时使用真实距离，否则用 P0 median。重点看各候选 `hit_rate`、`residual_median`、`residual_mad` 和 pairwise `median_diff_left_minus_right`，判断 P1/P2 是否只是稳定偏置，还是存在离群/低覆盖问题。

训练和评估入口:

```bash
python3 trajectory_fusion/train_reliability.py ~/mid360_datasets/trajectory/dataset_manifest.yaml \
  -o ~/mid360_datasets/trajectory/reliability_net.pt

python3 trajectory_fusion/evaluate_fusion.py ~/mid360_datasets/trajectory/traj_p0p1_001.csv \
  --json-out ~/mid360_datasets/trajectory/traj_p0p1_001.eval.json \
  --csv-out ~/mid360_datasets/trajectory/traj_p0p1_001.eval.csv
```
