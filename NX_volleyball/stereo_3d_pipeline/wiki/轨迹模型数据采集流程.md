# 轨迹模型数据采集流程

最后核对: 2026-07-07

轨迹模型训练使用固定配置 `config/pipeline_record_p0p1.yaml`。当前正式采集口径冻结 P0/P1 + 自研 CUDA NCC + XFeat: 主 CSV 记录 `z_roi_multi_point`、`z_roi_center_patch`、`z_roi_cuda_template_match`、`z_roi_neural_xfeat`，SuperPoint/ALIKED 在该配置中关闭。录制开启 `recording.detail_level: extended`、`raw_mode: true` 和 frame sidecar；diagnostic sidecar 默认关闭，只在临时 debug 配置中打开。训练读取原始候选字段，例如 `z_bbox_center`、`z_circle_center`、`z_roi_edge_centroid`、`z_roi_radial_center`、`z_roi_edge_pair_center`、`z_roi_multi_point`、`z_roi_center_patch`、`z_roi_cuda_template_match`、`z_roi_neural_xfeat`。不要把 `z_stereo` 或 `z` 当训练候选或标签。

## NX 启动

配置或代码更新后先构建一次:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
cmake --build build_standalone -j$(nproc)
```

每段数据使用递增文件名前缀，例如 `traj_p0p1_001`、`traj_p0p1_002`。录制前先写同名前缀的 metadata:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
mkdir -p ~/trajectory_dataset
nano ~/trajectory_dataset/traj_p0p1_001.metadata.yaml
```

metadata 示例:

```yaml
dataset_type: static
scene: center_3m
known_z: 3.0
static: true
gravity_y: 0.0  # 相机 y 轴未确认对齐重力前先写 0；确认后再写 9.81 或标定值
notes: static ball, center
```

启动录制:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
sudo jetson_clocks
cd build_standalone
./stereo_pipeline \
  --config ../config/pipeline_record_p0p1.yaml \
  --recording-out ~/trajectory_dataset/traj_p0p1_001.csv \
  2>&1 | tee ~/trajectory_dataset/traj_p0p1_001.log
```

需要抽样查看正式实时路径的算法 zoom 效果时追加低频 dump 参数。它默认后台写盘，输出 `frame_*.png` 和 `frame_*.json`; 正式长时间采集建议先小段验证再打开:

```bash
./stereo_pipeline \
  --config ../config/pipeline_record_p0p1.yaml \
  --recording-out ~/trajectory_dataset/traj_p0p1_001.csv \
  --debug-realtime-dump \
  --debug-realtime-dump-dir ~/trajectory_dataset/traj_p0p1_001.debug \
  --debug-realtime-dump-stride 100 \
  --debug-realtime-dump-max 30 \
  2>&1 | tee ~/trajectory_dataset/traj_p0p1_001.log
```

按 `Ctrl-C` 停止，程序会 flush CSV。下一段把文件名前缀递增为 `traj_p0p1_002`。每段输出:

```text
traj_p0p1_001.csv
traj_p0p1_001.frames.csv
traj_p0p1_001.metadata.yaml
traj_p0p1_001.log
traj_p0p1_001.p2_diagnostic.csv  # optional, only when diagnostic results are enabled
```

正式 `pipeline_record_p0p1.yaml` 默认不写 `.p2_diagnostic.csv`；`z_roi_cuda_template_match` 和 `z_roi_neural_xfeat` 已进入主 CSV。只有临时打开 `p2_diagnostic_results_enabled` 或 debug dump 时，运行时才会根据 `--recording-out ~/trajectory_dataset/traj_p0p1_001.csv` 自动派生 `.p2_diagnostic.csv`。

## 采集内容

先采静态已知距离，再采动态。静态覆盖 `2m, 3m, 4m, 5m, 6m`，每段 60-120s，并在 metadata 写 `known_z`、`static: true`、场景位置。动态覆盖慢速前后移动、左右/上下移动、正常抛球/击球、遮挡或单侧漏检；没有真实距离时 `known_z: null`，但必须写清楚 `scene` 和 `notes`。

采集完成后只做最小确认: `log` 末尾应有 `TrajectoryRecorder: saved ... frames`，同名前缀下应有主 CSV 和 `.frames.csv`。

## P1 dy 回归

当修改 P0/P1 候选、pair dy prior、soft gate 或字段记录后，先在 NX 上跑短时回归。该脚本会使用正式 `pipeline_record_p0p1.yaml`，输出主 CSV、frames sidecar、P2 diagnostic sidecar、日志和字段统计报告:

```bash
cd ~/NX_volleyball/stereo_3d_pipeline
scripts/nx_p1_dy_regression.sh --build --duration 20
```

可选打开低频 zoom/debug:

```bash
scripts/nx_p1_dy_regression.sh --build --duration 20 \
  --debug-dump --debug-stride 100 --debug-max 20
```

验收重点:

- `p0p1_dy_center` 明显非零时，`z_roi_center_patch`、`z_roi_multi_point`、`z_roi_cuda_template_match` 不应因为固定 `dy=0` 大面积无效。
- 报告里的 `Circle source groups` 要单独看。`roi_fit/roi_fit` 以外的组里 `z_circle_center` 可能按设计无效，但 direct YOLO pair + GPU candidate 存在时，`z_roi_center_patch` 和 `z_roi_multi_point` 不应再被 circle seed 硬前置挡成全 0。
- `z_bbox_center`、`z_circle_center`、`z_roi_edge_centroid`、`z_roi_radial_center`、`z_roi_edge_pair_center` 应继续保持 P0 高覆盖。
- `p0p1_*_trust` 和 `p0p1_untrusted_mask` 应与候选有效/无效和 dy/disparity 聚类关系一致。
- `Stage2_AsyncRoiWorker`、`Stage2_DualYoloGpuCandidates`、`Stage2_CudaTemplateNccMatch` 仍需满足 100fps 预算；若字段恢复但 worker 长尾变大，先降 P2 频率或关 sidecar，再排算法本身。

## 本地验收和训练入口

下载后先跑离线数据质量检查。它会自动寻找同名前缀 metadata，例如 `traj_p0p1_001.csv` 对应 `traj_p0p1_001.metadata.yaml`:

```bash
cd NX_volleyball/stereo_3d_pipeline
python3 trajectory_fusion/check_dataset.py \
  ~/mid360_datasets/trajectory/traj_p0p1_001.csv \
  --json-out ~/mid360_datasets/trajectory/traj_p0p1_001.check.json
```

重点看 `rows/duration/fps`、`frame_gaps`、`frame_counter_delta/frame_number_delta`、P0/P1 主 CSV 有效率、median、MAD 和 `known_z_bias`。动态片段还要看 `source_breakdown`、`.frames.csv` 汇总、`depth_jump`，确认 direct pair、fallback L2R/R2L、候选深度跳变，以及 `cuda_template`、`neural_xfeat` 的有效率和耗时没有异常。静态已知距离数据如果 `known_z_bias` 明显偏离，应先复查标定、metadata 和同步状态，不要直接并入训练。

`evaluate_fusion.py` 和 batch suite 会额外输出动态运动学诊断列:

- `speed_rms_mps`
- `accel_z_rms_mps2`
- `accel_y_residual_rms_mps2`
- `ballistic_residual_rms_mps2`
- `motion_jerk_rms_mps3`

没有真实距离的抛球/移动片段，先用这些列比较 raw、`robust_smooth`、`robust_rts_smooth`、`depth_polyfit_smooth`、`calibrated_*` 和 `reliability_*` 的物理一致性。`depth_polyfit_smooth` 是离线 batch 多项式拟合上限 baseline，只用于伪真值候选和过平滑对照，不进实时路径。`gravity_y` 只在 metadata 明确写入时作为 y 轴重力残差目标；未确认相机 y 轴方向前写 `gravity_y: 0.0`，避免把错误重力轴当成模型优劣依据。

输出优先看 `workflow_report.md`，再查 `workflow_summary.json`、`manifest_validation.json`、`training_method_coverage.csv`、`training_feature_coverage.csv`、根目录 `candidate_consistency.csv`、`baseline_suite/suite_metrics.csv` 和 `baseline_suite/<clip>/candidate_consistency.csv`。报告顶部 `readiness` 用于快速判断当前数据集状态：`ready_for_sweep` 表示可以开始训练/比较，`ready_for_model_selection` 表示 sweep 已跑且顶部模型通过审计，`smoke_only/blocked` 则先按原因补数据或修字段。报告里的 `Configured Method Coverage` 只列本次 sweep 实际启用的方法，正式配置下应只出现 `p0p1_ncc_xfeat` 的 9 路候选；`Key Method Coverage` 是全量关注表，用于查看其它历史/实验字段是否有覆盖。`Candidate Consistency` 和 `Known-Z Bucket Candidates` 会列出全局/距离桶内候选深度的低残差高覆盖方法，细查再打开根目录 `candidate_consistency.csv`。正式首轮模型选择用 `trajectory_fusion/configs/sweep_known_distance_selection.json`；动态片段覆盖足够后再跑 `trajectory_fusion/configs/sweep_dynamic_regularization.json` 做第二轮探索。这两个正式配置已固定 `p0p1_ncc_xfeat`，不会默认读入 SuperPoint/ALIKED/GFTT 等历史字段；其它候选集合用方法消融单独比较。多距离静态数据默认用 `--stratify-known-z`，让每个已知距离尽量同时覆盖 train/val；要验证未见距离泛化时，用 `--holdout-known-z 4.0 --holdout-split val` 把某个距离桶整体留出。workflow 默认只用 `known_z` 做校准/评估/训练标签，不会把它作为 smoother update；除非明确要生成标签辅助伪真值，否则不要加 `--use-static-known-z`。

基础 known-distance workflow 干净后，再跑方法消融，判断是否应该把 NCC/XFeat 纳入最终输入:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/run_dataset_workflow.py \
  ~/mid360_datasets/trajectory \
  -o ~/mid360_datasets/trajectory/workflow_method_ablation_YYYYMMDD \
  --configs trajectory_fusion/configs/sweep_method_ablation.json \
  --val-ratio 0.2 \
  --stratify-known-z \
  --seed 7 \
  --gravity-y 0
```

这个配置逐步打开 `p0`、`p0p1`、`p0p1_ncc`、`p0p1_ncc_xfeat`。`methods` allowlist 会贯穿训练、direct consensus、robust/calibrated smoother 和 reliability smoother；训练数组里被禁方法的专属 depth/disparity/support/confidence/top5 特征也会置零。`sweep_variant_ranking.csv` 的 `method_allowlist` 列会标明每行实际启用的方法。只从 `ready_for_model_selection=true` 且 `top_decision=recommended` 的结果继续比较 heldout `known_z` bias/MAD；无 `known_z` 历史片段只能证明配置可跑。

如果要一次跑完常规 known-distance、逐距离 holdout 和动态正则矩阵，用:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/run_workflow_matrix.py \
```

矩阵输出 `workflow_matrix_summary.json`、`workflow_compare.csv` 和 `workflow_compare.json`。当 known_z 桶少于 2 个时不会生成 holdout workflow。`--include-method-ablation` 会额外生成 `method_ablation` workflow，统一进入横向比较表。

如果同一天跑了常规 known-distance、leave-one-distance-out 和 dynamic 多组 workflow，用 `compare_workflows.py` 汇总:


只从比较表中 `ready_for_model_selection=true` 且 `top_decision=recommended` 的 workflow 继续看 checkpoint；方法消融结果要同时看 `top_method_allowlist` 和 `best_variant_method_allowlist`，确认排名对应的候选集合；`smoke_only` 或 `reject/no_known_z` 只能说明工具链可跑。

多段数据用 manifest 管理。采集目录里有多段主 CSV 后，先生成 manifest:

```bash
cd NX_volleyball/stereo_3d_pipeline
.venv-stereo-neural/bin/python trajectory_fusion/build_dataset_manifest.py \
  ~/mid360_datasets/trajectory \
  -o ~/mid360_datasets/trajectory/dataset_manifest.yaml \
  --val-ratio 0.2 \
  --stratify-known-z \
  --seed 7
```

生成器只收 TrajectoryRecorder 主 CSV，会跳过 `.frames.csv`、`.p2_diagnostic.csv`、`suite_metrics.csv`、`robust_smooth.csv` 等 sidecar/评估派生 CSV，并自动配对同名前缀 `*.metadata.yaml/.yml`。默认 `auto` split 会把带 `known_z` 的片段按比例分到 `train/val/test`，无 `known_z` 片段默认进 `train`；`--stratify-known-z` 会按毫米级 known_z 桶分别切分，避免某个距离被随机完全分到一个 split。`--holdout-known-z` 会覆盖随机/分层切分，把指定距离桶全部放到 `val` 或 `test`，其余已知距离进 `train`，用于 leave-one-distance-out 泛化检查。如果只是临时评估未标注片段，可加 `--unlabeled-split eval`。

manifest 文件路径相对 manifest 所在目录:

```yaml
clips:
  - csv: traj_p0p1_001.csv
    metadata: traj_p0p1_001.metadata.yaml
    split: train
  - csv: traj_p0p1_002.csv
    metadata: traj_p0p1_002.metadata.yaml
    split: val
```

训练前先做 manifest 级预检:

```bash
cd NX_volleyball/stereo_3d_pipeline
.venv-stereo-neural/bin/python trajectory_fusion/validate_dataset_manifest.py \
  ~/mid360_datasets/trajectory/dataset_manifest.yaml \
  --json-out ~/mid360_datasets/trajectory/dataset_manifest.check.json \
  --require-stratified-known-z
```

重点看 `split_counts`、`known_z_counts`、`known_z_bucket_counts`、`warning_counts` 和每段 `p0_min_hit_rate`。正式模型选择至少应有 `val` split；静态已知距离片段应覆盖 train/val，否则只能验证抖动趋势，不能用绝对 bias/MAD 选模型。`--require-stratified-known-z` 会提示具体哪个已知距离桶缺 train 或 val；如果刻意做某个距离只作为 heldout 的实验，不加这个开关。

训练前还要看模型实际输入覆盖:


这个检查会按 `dataset.py` 真实训练路径构造数组，确认 `METHOD_COLUMNS` 的候选有效率、top5/trust/pair gate 特征非零率，以及是否误把 legacy `z_stereo/z` 或在线 `x/y/z/v/a` 引回训练输入。`run_dataset_workflow.py` 已默认生成同名审计文件。

然后跑候选一致性诊断:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/analyze_candidate_consistency.py \
  ~/mid360_datasets/trajectory/dataset_manifest.yaml \
  --reference auto \
  --json-out ~/mid360_datasets/trajectory/candidate_consistency.json \
  --csv-out ~/mid360_datasets/trajectory/candidate_consistency.csv \
```

`--reference auto` 有 `known_z` 时使用真实距离，否则用 P0 median。重点看各候选 `hit_rate`、`residual_median`、`residual_mad` 和 pairwise `median_diff_left_minus_right`，判断 P1/P2 是否只是稳定偏置，还是存在离群/低覆盖问题。

明天多距离静态数据优先看 workflow 根目录 `candidate_consistency.csv` 的 `scope=known_z_bucket` 行。它按 `split + known_z_bucket` 汇总每个距离桶的候选误差，可以直接发现某个方法在 3m 稳、5m 漂，或 train/val 的同一距离偏置不一致。
`run_evaluation_suite.py` 默认也会为每个 clip 生成同样的候选一致性文件；上面的命令用于训练前快速复查或需要单独指定 reference 的情况。


这样 `sweep_metrics.csv` 会同时包含 `robust_rts_smooth`、`depth_polyfit_smooth`、`calibrated_smoother`、`calibrated_rts_smoother`、`reliability_direct`、`reliability_smoother` 和 `reliability_rts_smoother`。先看 `sweep_variant_ranking.csv` 判断神经模型是否真的超过固定 bias/sigma 校准、RTS 和 polyfit 离线上限 baseline，再看 `sweep_ranking.csv` 在 ReliabilityNet 内部选最佳 checkpoint。然后看 `sweep_reliability_methods.csv`，按 `config/variant/method` 检查 `valid/top_count/mean_sigma/mean_bias/mean_inlier_prob`，确认模型没有把权重异常集中到低覆盖或明显偏置的方法。最后看 `sweep_reliability_method_audit.csv` 的 `warnings` 和 `sweep_model_selection.csv` 的 `decision`，有 `reject` 或 `dominant_method_top_share` 的模型需要人工复查后再进入候选。
