# 实时管线

## 入口

实时入口在 `src/main.cpp`。核心调度已拆到 `src/pipeline/pipeline_loops.cpp`，异步 ROI Stage2 资源和 worker 在 `src/pipeline/pipeline_async_roi.cpp`，Stage0/1/2/3 主体和候选算法仍在 `src/pipeline/pipeline.cpp`。命令行参数包括:

```text
--config <path>
--visualize
--debug-feature-matches
--debug-feature-matches-dir <dir>
--recording-out <csv>
--record-baseline-clip
--baseline-out <dir>
--baseline-duration <sec>
--baseline-frames <n>
--baseline-clips <n>
--baseline-gap <sec>
--baseline-format <png|pgm>
--baseline-image-mode <gray|bgr|both>
--baseline-start-immediately
```

详细调度、同步点、丢旧帧策略和 profile 名称见 [实时调度细节](实时调度细节.md)。P2 特征候选进入 100fps 前的 lane 拆分、GPU 化和回传优化见 [架构调度优化](架构调度优化.md)。配置字段见 [配置参考](配置参考.md)，配置文件用途见 [配置矩阵](配置矩阵.md)。

`--debug-feature-matches` 是 legacy CPU 单帧 debug 入口，不输出 P2 GPU/VPI/TRT 后端的算法级左右点对。实时 P2 图像口径见 [P2算法效果与可视化审查](P2算法效果与可视化审查.md)。

## 运行合同

100fps 模式需要同时满足:

- 相机硬触发保持 100Hz。
- 左右帧水印 `frame_counter_delta` 稳定为 0。
- 取流队列可吸收 USB 到达抖动，但不能无限堆积。
- 检测、深度、记录任一阶段变慢时，优先丢弃旧后处理结果，不阻塞采集。
- 轨迹 CSV 和基准图像写盘使用队列限流。

## 配置要点

`config/pipeline_dual_yolo_roi.yaml` 中与实时性直接相关的字段:

| 字段 | 作用 |
|---|---|
| `camera.trigger_frequency_hz` | 相机触发频率 |
| `camera.image_node_num` | SDK FIFO 深度，用于吸收 USB 抖动 |
| `camera.embedded_info_clear_rows` | 清除水印行，避免影响后续图像处理 |
| `detector.dual_yolo.enabled` | 左右相机 YOLO 检测 |
| `detector.dual_yolo.depth_modes` | 深度候选开关 |
| `performance.drop_stale_roi_frames` | 后处理落后时丢旧帧 |
| `performance.async_roi_stage2` | 检测帧 ROI/特征/深度 Stage2 异步执行，下一帧 YOLO ready 后过期旧结果 |
| `performance.async_roi_buffers` | async ROI worker 自有图像快照 buffer 数 |
| `performance.async_roi_deadline_ms` | async ROI worker 软预算，用于 profile 和过期统计 |
| `recording.max_queue_frames` | CSV 写入队列上限 |
| `baseline_recording.max_queue_frames` | 基准片段写盘队列上限 |

## 基准片段录制

`--record-baseline-clip` 会把实时管线切到 detection-only, 关闭深度、tracker、神经特征和 ROS2, 只保留左右 YOLO 检测和校正图录制。输出目录为:

```text
baseline_clips/
  clip_YYYYMMDD_HHMMSS_01/
    left/
    right/
    left_bgr/        # image_mode=both 时存在
    right_bgr/       # image_mode=both 时存在
    frames.csv
    metadata.yaml
```

默认在检测到左/右球框后开始录制一段。录制阶段先把校正图按 `--baseline-image-mode gray|bgr` 缓存到内存，完成后再写盘；`both` 仅用于调试兼容。需要无检测 gate 的连续图像时使用 `--baseline-start-immediately`;需要多段时使用 `--baseline-clips` 和 `--baseline-gap`。

## 默认开启和关闭

100fps 默认适合开启:

- bbox/circle 几何候选
- ROI 边缘/径向轻量候选
- P1 CUDA multi-point / center patch 候选记录
- P1 sidecar 候选记录: CUDA Template/NCC、XFeat、SuperPoint
- GPU candidate refine
- epipolar fallback
- async ROI Stage2（需要重特征 A/B 时）
- 异步 CSV 记录

默认应保持关闭，除非做离线或专项 profile:

- ORB/BRISK/AKAZE/SIFT 类描述子
- 颜色 IoU patch 和其他非默认 patch A/B
- CPU fallback template
- 神经特征 realtime/inline 主路径
- 高成本可视化输出

## 实时输出和训练候选

实时 Stage2 会为每个有效左右 pair 计算多路原始 `z_*` 候选，并写入 `Object3D`/CSV。为了兼容 ROS、日志和旧轨迹预测，pipeline 仍会调用 `buildDepthCandidateObservations()`，再由 `selectLegacyDepthOutputCandidate()` 写入 `z_stereo`、`obj.z` 和 `stereo_depth_source`。

这个 legacy 输出只允许 P0 几何/bbox 和退化 fallback，不会让 P1 `z_roi_multi_point` / `z_roi_center_patch` / NCC / XFeat / SuperPoint 抢占在线主深度。训练可靠性模型时应读取各个原始候选字段，例如 `z_circle_center`、`z_roi_radial_center`、`z_roi_edge_pair_center`、`z_bbox_center`、`z_roi_multi_point`、`z_roi_cuda_template_match`、`z_roi_neural_xfeat`、`z_roi_neural_superpoint` 等；不要把 `z_stereo` 或 `z` 当作候选方法或标签。

## 增加新实时算法前的检查

1. 在基准片段上离线跑连续帧评估。
2. 输出有效点数、validation pass rate、深度中位数、MAD、帧间抖动、耗时和运动模型残差。
3. 在 NX 上确认 GPU/CPU 拷贝次数、CUDA stream 同步点、CPU 占用和写盘队列。
4. 默认配置保持关闭，只通过显式字段打开。
5. 更新 [深度测量](深度测量.md)、[性能测试](性能测试.md) 和本页。
