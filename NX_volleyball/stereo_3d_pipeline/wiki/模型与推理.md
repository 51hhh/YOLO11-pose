# 模型与推理

## 生产推理路径

实时检测使用 TensorRT engine。配置入口:

```yaml
detector:
  engine_path: "/home/nvidia/NX_volleyball/model/yolo26_gpu_fp16.engine"
  input_size: 640
  input_format: "bgr"
  use_dla: false
  dual_yolo:
    enabled: true
```

`detector` 和 `neural_feature_matching` 的完整字段说明见 [配置参考](配置参考.md)。构建目标和条件依赖见 [构建目标与依赖](构建目标与依赖.md)。

模型和引擎结论必须随配置一起记录:

- 模型文件名和训练数据版本。
- engine 构建命令。
- 精度模式: FP16、INT8、DLA/GPU。
- NX 上 `trtexec` 延迟和全管线 fps。
- 是否启用左右双 YOLO。

## 神经特征匹配

DISK、SuperPoint、LightGlue、XFeat 等神经特征匹配模型先作为离线验证和专项实时优化候选，不应直接进入默认 100fps 路径。当前本地 probe 已覆盖 XFeat、ALIKED、SuperPoint+LightGlue；DISK 尚未接入本地 probe 或 C++ 实时后端。

进入实时路径前必须满足:

- ROI 小图推理和匹配总耗时在目标预算内。
- 输出 10 个左右稳定关键点时 validation pass rate 明显优于传统方法。
- TensorRT 或 CUDA 路径明确，避免 Python/Torch 运行时进入生产程序。
- 输入、输出、归一化、坐标反变换和显存生命周期写入文档。

当前 C++ TensorRT 神经特征路径、fused engine 输入输出限制和离线 Python probe 见 [特征匹配实现](特征匹配实现.md)。

XFeat extractor-only engine 可通过 `scripts/export_xfeat_extractor_onnx.py` 和 `scripts/nx_build_xfeat_extractor_engine.sh` 构建；该路径当前仍会在 C++ 后处理阶段回到 CPU 做关键点筛选和匹配。ALIKED/SuperPoint 等若导出固定 shape `keypoints/descriptors/scores`，C++ 可走 direct extractor fallback 做真实 descriptor mutual-NN；`use_lightglue=true` 时 `matcher_engine_path` 支持固定 schema split matcher，但复杂动态 LightGlue schema 仍建议优先导出 fused 点对 engine。

## 推荐管理方式

- `scripts/` 放 engine 构建和部署脚本。
- `tools/` 放本机 CPU/GPU 验证工具。
- `config/` 放运行时是否启用对应模型的开关。
- Wiki 只写当前推荐路径和验证过的替代路径。
- 历史模型报告迁移到 [文档迁移索引](文档迁移索引.md)。

SOT tracker 的 NanoTrack/MixFormer engine、YOLO 帧间补框和 ROI 深度复用路径见 [追踪与补帧](追踪与补帧.md)。
