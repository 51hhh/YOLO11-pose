# 系统架构

## 数据流

```text
PWM 100Hz
  |
  v
海康左/右相机
  |
  v
同步抓帧 + FrameSpecInfo 水印
  |
  v
Bayer/Gray 缓冲 -> 校正/去畸变 -> 左右检测
  |
  v
左右 YOLO 框配对
  |
  v
ROI 深度候选 + 特征点验证
  |
  v
错误点剔除 + 视差融合
  |
  v
三角测距 + 轨迹记录/预测
```

## 模块职责

| 模块 | 职责 | 性能要求 |
|---|---|---|
| `capture` | 相机打开、触发、取流、水印读取 | 不阻塞 100Hz 取流 |
| `rectify` | VPI/CUDA 校正 | 保持 GPU/VPI 路径 |
| `detect` | TensorRT 预处理和 YOLO 推理 | 与采集/后处理并行 |
| `pipeline` | 帧槽、调度、丢弃策略、显示/记录线程 | 后处理落后时不拖慢采集 |
| `stereo` | ROI 深度候选、特征点匹配、双目验证 | 生产路径启用轻量方法 |
| `fusion` | 视差到深度、候选融合、轨迹预测 | 输出可记录、可训练 |
| `utils` | CSV 记录、基准片段录制、日志 | 写盘异步、队列限流 |

构建目标、条件依赖见 [构建目标与依赖](构建目标与依赖.md)，逐文件职责见 [代码地图](代码地图.md)。实时调度中的 Stage0/Stage1/Stage2、CUDA event 和旧 ROI 后处理丢弃条件见 [实时调度细节](实时调度细节.md)。

## Pipeline 源码分层

`src/pipeline/` 当前按调度、异步资源和算法主体拆分:

| 文件 | 边界 |
|---|---|
| `pipeline_loops.cpp` | `pipelineLoop()` / `pipelineLoopROI()` 主调度循环、FPS 统计、同步/异步 Stage2 提交点 |
| `pipeline_async_roi.cpp` | async ROI buffer 分配、gray/BGR/host snapshot、worker 线程、完成队列、过期和 buffer 回收 |
| `pipeline.cpp` | 初始化/启动/停止、Stage0/1/2/3 主体、dual YOLO 配对、ROI 候选、特征匹配、融合回调 |
| `pipeline.h` | `Pipeline` 私有接口、FrameSlot/async ROI task/result 结构和共享状态 |

异步 Stage2 的设计边界是: worker 只能在自有 snapshot buffer 上运行 `runRoiStage2Core()`，不能长期持有 `FrameSlot` 图像指针，也不直接更新 `HybridDepthEstimator` 或调用 callback。未过期结果由 pipeline 主线程 drain 后再 `applyRoiStage2Output()`，保证轨迹滤波和外部输出不会被 worker 乱序写入。

## 时序原则

实时系统按“最新有效观测”工作。采集、检测、后处理、写盘不能串成单线程长链路。

- 采集必须优先保持触发频率和同步稳定。
- 检测完成是后续 IoU、关键点、深度候选的自然边界。
- 若旧帧后处理落后，而新帧检测已经完成，应允许丢弃旧帧后处理，控制端到端延迟。
- 写盘、可视化和离线调试输出必须走异步队列或一次性调试入口。

## GPU 友好原则

- 图像预处理、ROI 候选、视差批处理优先留在 CUDA/VPI/TensorRT 路径。
- 不在 100fps 主路径中反复做 GPU/CPU 往返拷贝。
- OpenCV CPU 描述子、Python 神经匹配、可视化绘图默认属于离线评估路径。
- 如果必须在实时路径加新算法，先用基准片段离线验证，再在 NX 上做 profile，最后才打开配置。
