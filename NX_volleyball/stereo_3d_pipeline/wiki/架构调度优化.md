# 架构调度优化

最后核对: 2026-07-04

本页记录 P0/P1 不足以覆盖动态排球时，P2 特征/patch/dense stereo 候选要进入 100fps 实时路径所需的架构优化。当前目标不是把所有 P2 算法叠加到默认配置，而是把实时路径改到允许少量 P2 在 `10ms` 帧周期内稳定产出，同时保留低频诊断路径验证更重算法。

## 当前瓶颈

现有 `async_roi_stage2` 已经把重 ROI 后处理移出主 pipeline 线程，但检测帧结果仍受下一帧 YOLO ready 的 deadline 约束。`Stage2_AsyncRoiWorker` 超过 `async_roi_deadline_ms` 后，结果可能 stale/drop；FPS 仍可能接近 100，但候选有效率会下降。

当前 P2 的主要结构性成本:

- ROI Stage2 只有一个 worker，`runRoiStage2Core()` 外还有 `roi_postprocess_mutex_`，多候选会串行叠加。
- `DualYoloDepthGpuMatcher` 和 OpenCV CUDA P2 路径都会同步等待 GPU 结果并回到 CPU 聚合。
- OpenCV CUDA Template/BM/SGM 当前按小 ROI 调用库函数，但仍下载 score/disparity map 到 CPU 做 peak/robust 聚合。
- async ROI 目前以整帧 gray/BGR snapshot 为边界；P2 实际只需要小 ROI 或极线窗口时，整帧 D2D/BGR snapshot 会放大 copy 和 buffer 压力。
- CPU fallback、CPU OpenCV BRISK/AKAZE/SIFT 和 host gray D2H 会污染实时 P2 profile。
- debug zoom、ROS2 diagnostic、frame callback 需要图像时会引入额外 GPU/CPU 拷贝，不能和 realtime 准入混测。
- GPU 检测、CUDA rectify、P1 multi-point、P2 OpenCV CUDA 如果共用同一段时间窗口，可能出现 GPU stream 争用和尾延迟。
- 正常高置信 P0/P1 direct pair 帧不一定需要 P2；如果每帧全跑 P2，会把预算浪费在低收益帧上。

## 目标架构

把 Stage2 拆成两条 lane:

| lane | 用途 | deadline | 输出 |
|---|---|---:|---|
| realtime lane | P0/P1 + 最多少量轻量 P2，参与实时记录和后续训练候选 | 必须在下一帧 deadline 内完成 | `Object3D`、CSV 原始候选、frame sidecar |
| diagnostic lane | ORB/SGM/XFeat/SuperPoint/调参 sweep，可迟到或低频运行 | 可放宽或按 stride 运行 | debug JSON/zoom、候选 CSV、离线报告 |

Realtime lane 的 P2 准入建议: 单算法 `avg < 2-3ms`、`p95 < 5ms`、`max < 10ms`、`Stage2_AsyncRoiOverDeadline` 接近 0。Diagnostic lane 不允许反向阻塞 realtime lane，也不参与 legacy `z_stereo/obj.z` 选择。

## GPU 化和回传策略

P2 如果要有 100fps 机会，应尽量进入自研 CUDA batch 路径，而不是每个候选单独调用 OpenCV CUDA 后再下载中间图:

- 对 Template/NCC、ring/edge profile、窄带 dense disparity，优先写成 batch kernel。输入是一批左右 pair 和 ROI 参数，输出每个候选的 `disparity/support/stddev/confidence/valid`。
- GPU 内完成 peak search、second peak gap、MAD/RANSAC 或 robust aggregation，只 D2H 最终结构体。
- debug 模式才下载 score map、disparity map、keypoint/match 可视化数据。
- P2 与 P0/P1 共用 rect gray/BGR GPU snapshot，但避免为每个算法重复 copy ROI。
- 对神经特征，优先把 NMS、descriptor sampling、mutual NN 和几何 gate 移到 GPU 或 fused TensorRT engine，减少 CPU postprocess 和大 tensor D2H。
- 固定的自研小 ROI 流程可以评估 CUDA Graph: `ROI pack -> candidate kernels -> reduce -> D2H result`。OpenCV CUDA 调用不一定适合 graph capture，先用于自研 kernel。
- 对 Template/BM/SGM 的 OpenCV CUDA 版本，先用 profile 判断中间 map 下载是否主导；若主导，直接转自研 kernel，避免在库调用外继续堆调度优化。

## 选择性触发

P2 不应默认每帧全跑。Realtime lane 可以只在下列条件触发一个或少数 P2 候选:

- P0/P1 候选之间分歧超过阈值。
- direct pair 的 `pair_shifted_iou`、`pair_score`、`pair_bbox_prior_penalty` 或 `pair_epipolar_dy` 质量变差。
- 当前帧进入 fallback L2R/R2L，或 fallback 与轨迹预测差异较大。
- 选中深度相对上一帧/运动模型出现异常跳变。
- 训练采样需要按 stride 捕获少量 P2 候选，而不是全帧实时融合需要。

这样可以把 P2 预算集中在真正有不确定性的帧上。未触发 P2 的帧仍记录 P0/P1，触发状态和未触发原因应进入 frame sidecar 或调试摘要，避免把“未尝试”误读为“算法无效”。

## ROI pack 和 snapshot

后续 snapshot 可以从整帧复制逐步改为分层:

- realtime lane: 只复制或引用必要的 gray ROI / BGR ROI / 极线窗口。
- diagnostic lane: 需要 zoom、score map 或人工检查时才保留整帧或大 ROI。
- BGR snapshot lazy copy: 只有颜色 patch、BGR 神经特征或 debug 需要时才复制。
- ROI pack 输出连续内存，给自研 CUDA batch kernel、OpenCV CUDA view 或 TensorRT ROI 输入复用。
- 对单帧多 pair，先 pack 所有 ROI，再一次 batch 计算，避免每个 pair 单独 copy 和 launch。

## OpenCV CUDA 并行化边界

多个 OpenCV CUDA 算法可以尝试并行，但不能假设库调用天然轻量:

- 每个算法使用独立 `cv::cuda::Stream`、独立 scratch buffer、独立 matcher 实例，避免 thread-local scratch 互相覆盖。
- 预创建 `TemplateMatching`、`StereoBM`、`StereoSGM`、ORB extractor/matcher，避免每帧创建对象。
- 避免 `download()` 中间 map；若必须下载，只在 diagnostic lane 或 stride debug 中做。
- 实测是否存在隐式同步。若 OpenCV CUDA 内部同步或分配开销导致尾延迟，realtime lane 应改为自研 kernel。
- 多流并行只在 GPU 有空闲时有效；如果 YOLO 使用 GPU 而不是 DLA，P2 并行可能恶化检测尾延迟。
- OpenCV CUDA 并行化需要单独 A/B: 单算法单 stream、多算法同 stream、多算法多 stream、预创建对象和每帧创建对象。准入看 p95/p99，不只看 avg。

## Worker 和锁优化

后续实现顺序:

1. 把 `roi_postprocess_mutex_` 缩小到确实非线程安全的 matcher/scratch 对象；纯数据整理和独立 matcher 不应持有全局锁。
2. 将 P2 拆成 `FeatureJob`，独立于完整 `runRoiStage2Core()`。FeatureJob 持有小 ROI buffer、frame id、left/right pair id 和 deadline。
3. realtime lane 保留一个低延迟 worker；diagnostic lane 使用独立队列和低优先级 stream，可按 stride 或 max-in-flight 限流。
4. async buffer 从整帧 snapshot 逐步优化为“整帧基础快照 + 小 ROI job view/ROI copy”，避免为低频 debug 长期持有 FrameSlot。
5. 对 fallback 场景单独建 job 类型。direct pair P2 和 fallback epipolar P2 不应混用字段或 gate。

## Profile 准入

P2 realtime 复测必须同时看:

- `Stage2_AsyncRoiWorker` avg/p50/p90/p95/p99/max。
- `Stage2_AsyncRoiOverDeadline`、`Stage2_AsyncRoiDrop*`、`Stage2_DropStaleROI`。
- 算法级 profile 的 avg/p95/max，例如 `Stage2_OpenCVCudaTemplateMatch`、`Stage2_OpenCVCudaStereoBM`、`Stage2_OpenCVCudaStereoSGM`、`Stage2_NeuralFeatureMatch`。
- `Stage2_AsyncRoiNeedHostGray` 和 `Stage2_AsyncRoiHostGrayD2HSubmit` 必须为 0，除非本次明确测试 CPU fallback。
- P2 attempted / not_attempted / valid / reject reason，区分“未触发”“触发但无效”“触发且过期”。
- frame sidecar 的 direct/fallback 数量、候选有效率、median/MAD、帧间跳变。
- debug/diagnostic 开关必须单独测，不和 realtime 准入混在一起。

## 融合边界

P2 在进入默认融合前要先作为原始候选记录。Realtime lane 里按时完成的 P2 可以写 CSV 和训练候选；是否更新 `HybridDepth` 应由可靠性和延迟共同决定。Diagnostic lane 的迟到结果不能直接回写 `HybridDepth`，只能按 `frame_id` 写候选、debug 或离线报告。

## 风险清单

- 只看 FPS 会误判。P2 stale/drop 后 FPS 仍可能接近 100，但候选字段有效率下降。
- 单项 isolated 通过不代表能叠加到 P0+P1 默认配置；P1 multi-point 已经占用明显预算。
- OpenCV CUDA 小 ROI 可能被固定调用开销、临时分配和 D2H 主导，ROI 变小不一定线性变快。
- 多流并行可能提高吞吐但增加尾延迟；100fps 更关心 p95/max。
- Diagnostic lane 迟到结果不能直接更新 HybridDepth，只能按 frame id 落候选或 debug。
- CPU fallback 自动介入会让“GPU P2”测试结论失真。
- 选择性触发会改变分母；报告中必须同时给 attempted 和 total frames，否则会高估有效率。
