# P0/P1 + 三算法联合测试 2026-07-05

最后核对: 2026-07-05

本页记录 2026-07-05 对当前实时管线的联合验证: P0/P1 主线保持 100Hz，同时在 sidecar 中并行采集自研 CUDA Template/NCC、XFeat TensorRT、SuperPoint TensorRT。三算法现在按 P1 训练候选管理，但运行承载仍是 diagnostic worker，不回写主融合。该测试用于确认“多个训练候选同时开启时，主线是否仍稳定”和“训练读取字段是否独立”。

## 代码改动整理

本轮代码改动分为四类:

| 类别 | 文件 | 说明 |
|---|---|---|
| 神经配置拆分 | `src/main_config_loaders.cpp`, `src/pipeline/pipeline_config.h` | `neural_feature_matching` 保留 legacy 单后端；新增 `neural_feature_matching_xfeat` 和 `neural_feature_matching_superpoint`，分别写独立配置 |
| 双 matcher 实例 | `src/pipeline/pipeline.cpp`, `src/pipeline/pipeline.h` | 新增 `neural_xfeat_matcher_`、`neural_superpoint_matcher_`；两个 TensorRT matcher 可同帧顺序执行，不再互斥 |
| sidecar 路由 | `src/pipeline/pipeline_async_roi.cpp`, `src/pipeline/pipeline_feature_jobs.cpp`, `src/pipeline/pipeline_dual_yolo_match.cpp`, `src/pipeline/pipeline_roi_stage2.cpp` | 一个 diagnostic job 内可输出 `cuda_template`、`neural_xfeat`、`neural_superpoint` 三种 mode；主线仍不回写这些 sidecar 结果 |
| 训练字段合并 | `trajectory_fusion/dataset.py`, `trajectory_fusion/evaluate_fusion.py`, `trajectory_fusion/schema.md`, `trajectory_fusion/tests/test_synthetic_dataset.py` | 训练读取器按 `frame_id + mode` 合并 `z_roi_cuda_template_match`、`z_roi_neural_xfeat`、`z_roi_neural_superpoint`；新增 synthetic test 防止 sidecar mode 互相覆盖 |

修复项:

- baseline/detection-only override 现在会同时关闭 legacy `neural_features`、`neural_xfeat` 和 `neural_superpoint`。
- split matcher 如果配置启用但缺少 engine，会真正置 `enabled=false`，避免后续 P2 mask 写出无意义 `unavailable` 行。
- 神经 sidecar 不再使用额外外层 sparse geometry gate；XFeat/SuperPoint 自身仍有 mutual match、y/disp/MAD/spatial gate。

## 当前配置口径

正式采集配置:

```text
config/pipeline_record_p0p1.yaml
config/pipeline_dual_yolo_roi.yaml
```

当前默认是:

- P0/P1 主 CSV 每帧运行并记录。
- `roi_cuda_template_match: true`，作为 P1 sidecar 训练候选记录，不进入 legacy `z_stereo/obj.z`。
- `neural_feature_matching.enabled: false`，legacy 单后端关闭。
- `neural_feature_matching_xfeat.enabled: true`。
- `neural_feature_matching_superpoint.enabled: true`。
- `p2_realtime_lane_decision_enabled: false`。
- `p2_diagnostic_lane_decision_enabled: true`。
- `p2_diagnostic_stride: 1`，sidecar 每帧尝试采集。
- `p2_diagnostic_max_in_flight: 1`，上一帧 sidecar 未完成时直接丢弃新 sidecar，不阻塞 P0/P1。
- `p2_diagnostic_use_source_snapshot: true`，diagnostic worker 借用 async ROI snapshot，避免二次整帧 D2D copy。
- `roi_cuda_gftt_lk: false`，当前组合不再默认采集 GFTT/LK。

代码中还保留按算法单独降频的调度地基，当前正式 YAML 均设为 1:

| 配置 | 当前状态 | 作用 |
|---|---|---|
| `p2_diagnostic_stride_cuda_template` | `1` | >0 时覆盖 NCC 的 diagnostic stride |
| `p2_diagnostic_stride_neural_xfeat` | `1` | >0 时覆盖 XFeat 的 diagnostic stride |
| `p2_diagnostic_stride_neural_superpoint` | `1` | >0 时覆盖 SuperPoint 的 diagnostic stride |
| `p2_diagnostic_use_source_snapshot` | `true` | diagnostic worker 借用 async ROI snapshot，避免二次整帧 D2D copy |

## 当前工程结论

1. 当前可稳定使用的实时主线是 P0/P1 主 CSV 100Hz。
2. NCC、XFeat、SuperPoint 可以同时开启采集，当前按 P1 sidecar 训练候选管理。
3. 三个 sidecar 方法不参与 `z_stereo/obj.z`，训练通过 `trajectory_fusion/dataset.py` 合并 sidecar 字段。
4. SuperPoint 当前覆盖率最好，但深度相对 P0/P1 有系统差；需要可靠性模型学习偏置。
5. XFeat 有效帧不足，保留为辅助训练候选和后续优化目标。
6. NCC 是稳定单点模板候选，`support=1` 不能和多点方法直接比较。

## 后续排查入口

- 性能异常: 看 `Stage2_AsyncRoiWorker`、`Stage2_P2FeatureJobDiagnosticWorker`、`Stage2_P2FeatureJobDiagnosticDropFull`。
- sidecar 覆盖率异常: 看 `traj.frames.csv` 的 `p2_diagnostic_requested` 和 `traj.p2_diagnostic.csv` 的 `mode/status`。
- 主 CSV 看不到三算法字段: 当前是预期，确认 `*.p2_diagnostic.csv` 存在，并用训练 loader 合并。
- 如果要每帧完整三算法: 需要进一步优化神经后处理或启用独立按算法 stride，不能只把 `p2_diagnostic_stride` 改回 1。
