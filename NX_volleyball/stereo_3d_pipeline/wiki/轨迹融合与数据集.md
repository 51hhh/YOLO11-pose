# 轨迹融合与数据集

## 目标

轨迹融合模块记录多种深度候选、检测几何、同步状态和诊断字段，用于训练或评估可靠性模型。它不应掩盖实时深度算法的问题，而是帮助判断不同候选在不同条件下的可信度。

## 数据入口

实时记录配置:

```yaml
recording:
  enabled: true
  output_path: "dual_yolo_observation_data.csv"
  detail_level: "extended"
  raw_mode: true
  max_queue_frames: 1000
```

字段定义以 `trajectory_fusion/schema.md` 为准。

实时 `Object3D`、TrajectoryRecorder 和 baseline clip CSV 的字段关系见 [数据结构与Schema](数据结构与Schema.md)。深度候选字段分组见 [深度候选字段参考](深度候选字段参考.md)，实机采集流程见 [轨迹模型数据采集流程](轨迹模型数据采集流程.md)。

训练候选以 `trajectory_fusion/dataset.py` 的 `METHOD_COLUMNS` 为准，只包含原始 `z_*` 候选字段，不包含 legacy `z_stereo` 或在线 `z`。实时 C++ 为兼容 ROS、日志和旧轨迹预测仍会用 `buildDepthCandidateObservations()` + `selectLegacyDepthOutputCandidate()` 写出 `z_stereo/z`，但这只允许 P0 几何/bbox 和退化 fallback，不是训练模型的深度决策。

当前 100fps 采集口径以 P0 几何、P1 主 CSV `z_roi_multi_point` / `z_roi_center_patch` 和 P1 sidecar 三算法为主。`dataset.py` 会自动查找同名前缀 `*.p2_diagnostic.csv`，并按 `frame_id + mode` 合并 `cuda_template` -> `z_roi_cuda_template_match`、`neural_xfeat` -> `z_roi_neural_xfeat`、`neural_superpoint` -> `z_roi_neural_superpoint`，同时合并对应 disparity/support/std/confidence。OpenCV CUDA GFTT/LK、color/color-edge、SGM 和 VPI 字段保留为历史/实验候选，默认不启用。没有 sidecar 时这些候选保持无效。

本地静止历史片段的第一轮模型对比见 [轨迹模型本地实验20260707](轨迹模型本地实验20260707.md)。关键结论是: 没有 `known_z` 时，单独 ReliabilityNet 容易发生绝对深度漂移；当前最稳 baseline 是“候选方法分组 + 鲁棒物理 smoother”。ReliabilityNet 后续应输出每路 `bias/sigma/outlier_prob`，再交给 EKF/UKF/RTS 类物理滤波器，而不是直接当最终轨迹。

| `trajectory_fusion/compare_workflows.py` | 横向比较多个 workflow，输出 readiness、top selection、best variant 和 warning 汇总 |
| `trajectory_fusion/README.md` | 实验说明和历史结论 |

多段训练数据使用 `dataset_manifest.yaml` 管理。采集目录整理好后优先用工具生成:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/run_dataset_workflow.py \
  ~/mid360_datasets/trajectory \
  -o ~/mid360_datasets/trajectory/workflow_YYYYMMDD \
  --configs trajectory_fusion/configs/sweep_known_distance_selection.json \
  --val-ratio 0.2 \
  --stratify-known-z \
  --seed 7 \
  --gravity-y 0
```

它会写 `dataset_manifest.yaml`、`manifest_validation.json`、`training_input_audit.json`、`training_method_coverage.csv`、`training_feature_coverage.csv`、全局 `candidate_consistency.csv/json`、`method_calibration.json`、`baseline_suite/suite_metrics.csv`、`workflow_summary.json` 和 `workflow_report.md`；默认还会跑 ReliabilityNet sweep。`workflow_report.md/json` 的 `readiness` 会给出 `smoke_only`、`blocked`、`ready_for_sweep` 或 `ready_for_model_selection`，用于区分“只能冒烟”“数据可训练”和“模型已可选”。`--stratify-known-z` 会按每个已知距离桶独立切分 train/val/test，适合多距离静态数据做常规 heldout 选择；`--holdout-known-z 4.0 --holdout-split val` 会把某个距离桶整体留出，用于 leave-one-distance-out 泛化检查。临时字段 smoke 用 `--skip-sweep --skip-rts-smoother --skip-depth-polyfit`；PyTorch 链路 smoke 用 `--configs trajectory_fusion/configs/sweep_smoke.json --skip-rts-smoother --skip-depth-polyfit`。默认不把 `known_z` 作为 smoother update，只有生成标签辅助伪真值时才显式加 `--use-static-known-z`。

多组 workflow 跑完后用下面命令生成横向比较表:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/compare_workflows.py \
  ~/mid360_datasets/trajectory/workflow_known_YYYYMMDD \
  ~/mid360_datasets/trajectory/workflow_holdout_4m_YYYYMMDD \
  ~/mid360_datasets/trajectory/workflow_dynamic_YYYYMMDD \
  --csv-out ~/mid360_datasets/trajectory/workflow_compare_YYYYMMDD.csv \
  --json-out ~/mid360_datasets/trajectory/workflow_compare_YYYYMMDD.json
```

比较表按 `ready_for_model_selection`、`ready_for_sweep`、`smoke_only`、`blocked` 排序，并带出 top model decision、known-distance bias/MAD、best variant、候选一致性首项和 warning。最终只从 `ready_for_model_selection=true` 且 `decision=recommended` 的 workflow 里选模型。

如果要直接跑标准矩阵，用:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/run_workflow_matrix.py \
  ~/mid360_datasets/trajectory \
  -o ~/mid360_datasets/trajectory/workflow_matrix_YYYYMMDD \
  --include-dynamic \
  --gravity-y 0
```

矩阵会先跑 `known_stratified`，再为每个已知距离桶生成 `holdout_<distance>` workflow；只有存在至少两个 known_z 桶时才会跑 holdout。`--include-dynamic` 会额外跑 `sweep_dynamic_regularization.json`。输出目录下会有 `workflow_matrix_summary.json`、`workflow_compare.csv` 和 `workflow_compare.json`。

如果需要拆开排查，可按下面命令逐项执行。manifest 生成:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/build_dataset_manifest.py \
  ~/mid360_datasets/trajectory \
  -o ~/mid360_datasets/trajectory/dataset_manifest.yaml \
  --val-ratio 0.2 \
  --stratify-known-z \
  --seed 7
```

生成器只纳入 TrajectoryRecorder 主 CSV，跳过 `.frames.csv`、`.p2_diagnostic.csv` 和 suite/sweep 输出。默认 `auto` split 会优先把带 `known_z` 的片段分出 `val`，用于 heldout 排名。多距离静态采集应加 `--stratify-known-z`，避免某个距离只出现在 train 或 val；同一距离只有 1 段时工具会保留 train，不能替代补采 heldout 片段。要检查模型对未见距离的泛化，可改用 `--holdout-known-z <distance>`，指定距离全部进入 `--holdout-split`。

manifest 形状如下:

```yaml
clips:
  - csv: traj_p0p1_001.csv
    metadata: traj_p0p1_001.metadata.yaml
    split: train
  - csv: traj_p0p1_002.csv
    metadata: traj_p0p1_002.metadata.yaml
    split: val
```

路径相对 manifest 所在目录解析。`train_reliability.py` 默认只优化 `split: train` 的片段，其他 split 会作为 heldout 记录进 checkpoint metadata。
训练配置可把 `--leave-one-weight` 作为一个 sweep 维度；它会临时遮住某个有效测距方法，用其他候选预测被遮住方法的观测一致性，帮助模型学习“候选之间谁能互相解释”。

采集完成后先做 manifest 级预检:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/validate_dataset_manifest.py dataset_manifest.yaml \
  --json-out test_logs/trajectory_manifest_check_YYYYMMDD.json \
  --require-stratified-known-z
```

这个入口会逐段调用 `check_dataset.py`，同时汇总 `train/val/test` split、`known_z` 覆盖、FPS、帧号缺口、水印异常和 P0/P1 主字段命中率。正式训练前必须至少有 `val` split；静态已知距离数据还应覆盖 `train` 和 `val`，否则 sweep 排名只能按无标签稳定性退化排序。`--require-stratified-known-z` 会额外检查每个 known_z 距离桶是否都有 train/val 覆盖；如果刻意做 leave-one-distance-out 实验，可以不加这个开关。

训练输入审计:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/audit_training_inputs.py dataset_manifest.yaml \
  --json-out test_logs/training_input_audit_YYYYMMDD.json \
  --method-csv-out test_logs/training_method_coverage_YYYYMMDD.csv \
  --feature-csv-out test_logs/training_feature_coverage_YYYYMMDD.csv
```

这个入口复用 `build_legacy_arrays()`，因此审计的是 ReliabilityNet 实际看到的数组，不是单纯 CSV 表头。重点看 `training_method_coverage.csv` 里 P0/P1/NCC/XFeat 的 `hit_rate`，以及 `training_feature_coverage.csv` 里 top5、trust、水印、pair gate 字段是否长期为 0。`workflow_report.md` 会汇总帧数、方法数、特征数和审计 warning。

全局候选一致性文件位于 workflow 根目录，和 `baseline_suite/<clip>/candidate_consistency.csv` 不同。根目录 `candidate_consistency.csv` 会额外写 `scope=known_z_bucket` 行，用 `split + known_z_bucket` 汇总每个距离桶内各候选方法的 residual median/MAD/95% 误差；这用于判断某个方法是固定 bias，还是在 3m/4m/5m 等不同距离上出现距离相关偏置。

候选一致性诊断:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/analyze_candidate_consistency.py dataset_manifest.yaml \
  --reference auto \
  --json-out test_logs/trajectory_candidate_consistency_YYYYMMDD.json \
  --csv-out test_logs/trajectory_candidate_consistency_YYYYMMDD.csv \
  --pairwise-csv-out test_logs/trajectory_candidate_pairwise_YYYYMMDD.csv
```

`--reference auto` 在 metadata 有 `known_z` 时使用真实距离，否则退化为 P0 median。这个报告用于看每种候选的有效率、相对 bias/MAD、95% 残差，以及候选之间的共现偏置；它不能替代 `known_z` 绝对精度评估，但能提前发现 P1/P2 系统偏置和低覆盖字段。

明天新数据的首轮评估建议直接跑 batch suite。suite 默认会在每个 clip 目录写出 `candidate_consistency.json`、`candidate_consistency.csv` 和 `candidate_pairwise.csv`，因此单独候选诊断命令主要用于快速复查或自定义 reference:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/run_evaluation_suite.py dataset_manifest.yaml \
  -o test_logs/trajectory_model_eval_YYYYMMDD \
  --calibration test_logs/method_calibration_YYYYMMDD.json \
  --checkpoint test_logs/trajectory_model_experiments_20260707/reliability_static_h32_e120_j1_bias10.pt \
  --gravity-y 0
```

没有 checkpoint 或 calibration 时也可以运行同一脚本，它会跳过对应模型，只输出可用的 `check_dataset`、候选一致性、raw eval、`robust_smooth` 因果 baseline、`robust_rts_smooth` 离线 RTS baseline 和 `depth_polyfit_smooth` 离线多项式拟合 baseline。
注意: suite 默认不把 `known_z` 当作 smoother update，只用它在 `check_dataset`/`evaluate_fusion` 里计算 bias/MAD，避免已知距离标签泄漏到输出轨迹。只有要生成标签辅助伪真值时才显式加 `--use-static-known-z`。
`robust_rts_smooth`、`calibrated_rts_smoother`、`reliability_rts_smoother` 和 `depth_polyfit_smooth` 会使用全序列未来帧，只用于离线评估/伪真值候选；需要只看实时因果路径时加 `--skip-rts-smoother` / `--skip-depth-polyfit`，或只比较不带 `_rts_` 和 `polyfit` 的 variant。

suite 跑完后汇总核心指标:

```bash
python3 trajectory_fusion/summarize_evaluation_suite.py test_logs/trajectory_model_eval_YYYYMMDD \
  --methods-output test_logs/trajectory_model_eval_YYYYMMDD/suite_reliability_methods.csv
```

`suite_metrics.csv` 除 `z_std`、`known_z_bias` 外，还包含动态片段用的 `speed_rms_mps`、`accel_z_rms_mps2`、`accel_y_residual_rms_mps2`、`ballistic_residual_rms_mps2` 和 `motion_jerk_rms_mps3`。没有 `known_z` 的抛球/移动片段不能评估绝对深度，但可以用这些列判断轨迹是否物理一致。`gravity_y` 从 metadata 读取；未写时按 0 处理。

ReliabilityNet 结果不要只看 `z_std`。`evaluate_reliability_smoother.py --json-out` 的每个 sequence 里有 `method_summary`；`summarize_evaluation_suite.py --methods-output` 会把它整理成 `suite_reliability_methods.csv`，记录每个方法的 `valid`、`top_count`、`mean_sigma`、`mean_bias`、`mean_abs_bias`、`mean_inlier_prob`、`mean_raw_z`、`mean_corrected_z` 和 `mean_corrected_minus_raw_z`。明天看模型时优先检查三件事: `top_count` 是否集中到单一路径、`mean_bias` 是否只是把所有方法平移到一个无真值深度、低覆盖方法是否被给了异常小的 `mean_sigma`。`evaluate_reliability_checkpoint.py` 的 direct consensus 只做对照；其 `smooth_sigma_z` 现在来自融合权重，未训练的 common sigma 仅作为 `reliability_common_sigma` 保留。

可以直接审查方法级风险:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/audit_reliability_methods.py \
  test_logs/trajectory_model_eval_YYYYMMDD/suite_reliability_methods.csv \
  -o test_logs/trajectory_model_eval_YYYYMMDD/reliability_method_audit.csv
```

重点看 `dominant_top_share`、`low_coverage_top_share` 和 `warnings`。如果排名靠前的模型同时出现 `dominant_method_top_share`、`low_coverage_methods_receive_top_weight` 或 `low_coverage_methods_have_tiny_sigma`，先不要把它当作最佳模型。

也可以把排名和 audit 合并成选择表:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/select_reliability_model.py \
  test_logs/trajectory_model_sweep_YYYYMMDD/sweep_metrics.csv \
  --audit test_logs/trajectory_model_sweep_YYYYMMDD/sweep_reliability_method_audit.csv \
  -o test_logs/trajectory_model_sweep_YYYYMMDD/sweep_model_selection.csv
```

`decision=recommended` 表示指标排序靠前且没有严重方法风险；`caution` 表示缺 audit 或只有偏置类风险；`reject` 表示出现权重集中到低覆盖/低 inlier/低 sigma 方法等严重风险。

需要批量训练多个 ReliabilityNet 配置时:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/run_reliability_sweep.py dataset_manifest.yaml \
  -o test_logs/trajectory_model_sweep_YYYYMMDD \
  --calibration test_logs/method_calibration_YYYYMMDD.json \
  --gravity-y 0
```

默认 sweep 会覆盖 `hidden=32/64`、`epochs=80/120` 和一个低权重 leave-one 配置；也可以用 `--configs sweep.yaml` 提供自定义配置列表。
脚本会写出 `sweep_metrics.csv`、`sweep_ranking.csv`、`sweep_variant_ranking.csv`、`sweep_reliability_methods.csv`、`sweep_reliability_method_audit.csv` 和 `sweep_model_selection.csv`。`sweep_metrics.csv` 保留 manifest 的 `split`；传 `--calibration` 时每个 suite 也会包含 `calibrated_smoother` 和 `calibrated_rts_smoother` variant。`sweep_ranking.csv` 默认只排 `reliability_smoother`；`sweep_variant_ranking.csv` 等价于 `rank_sweep_metrics.py --variant all`，用于确认神经模型是否超过固定校准、`robust_smooth`、`robust_rts_smooth` 和 `depth_polyfit_smooth` baseline。排名默认 `--rank-split auto`，如果存在 `val` split 就只用 heldout 指标，避免按训练集表现选模型。有 `known_z` 的静态片段会优先按绝对 bias/MAD 排名，没有 `known_z` 时按 `z_std`、峰峰值和动态运动学残差联合排名。

非神经校准 baseline:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/fit_method_calibration.py dataset_manifest.yaml \
  -o test_logs/method_calibration_YYYYMMDD.json \
  --train-split train

.venv-stereo-neural/bin/python trajectory_fusion/evaluate_calibrated_smoother.py \
  test_logs/method_calibration_YYYYMMDD.json val_clip.csv \
  --metadata val_clip.metadata.yaml \
  --gravity-y 0 \
  -o test_logs/val_clip_calibrated_smooth.csv \
  --json-out test_logs/val_clip_calibrated_smooth.json
```

这个 baseline 不训练 GRU，只学习“每种方法相对真实距离的固定 bias/sigma”。它不能替代动态可靠性模型，但非常适合明天先回答两个问题: 哪些字段只是稳定偏置，哪些字段在 heldout 距离上仍然不稳定。`known_z` 只用于拟合 train split 的 calibration；评估输出仍不把 `known_z` 喂回 smoother。

`check_dataset.py` 除基础行数、FPS、水印和 P0/P1 主 CSV 深度统计外，还会输出:

- `source_breakdown`: direct pair、fallback L2R/R2L、circle source 和 depth source 组合计数。
- `frame_summary`: 自动读取同名前缀 `.frames.csv`，汇总每帧 raw/stereo/direct/fallback 数量。
- `depth_jump`: 各候选相邻有效帧深度跳变的 median/MAD/p95/max。
- `p2_diagnostic`: 自动读取同名前缀 `*.p2_diagnostic.csv`，按 `mode` 汇总 rows、valid、median/MAD、algo p95、worker p95 和 deadline 次数。

VPI sidecar 不回写主 CSV，但 `check_dataset.py` 会输出 sidecar 摘要；`dataset.py` 训练入口会把 `vpi_template_match`、`vpi_orb` 自动合并成候选字段。

## 历史90fps静止CSV本地实验

本地已用历史 `P0/P1 + XFeat + NCC` 静止片段跑过两组 ReliabilityNet sweep:

```bash
.venv-stereo-neural/bin/python trajectory_fusion/run_dataset_workflow.py \
```

候选一致性以 P0 median 为参考时，历史片段里最稳的是 `roi_edge_pair_center`、`circle_center`、`roi_radial_center` 和 bbox 相关字段；`roi_multi_point`、`roi_cuda_template_match`、`roi_neural_xfeat` 有稳定偏置或低覆盖，适合作为模型候选输入，不适合在无 `known_z` 下直接提升为主观测。

这组结果的工程含义:

- ReliabilityNet 结构本身可训练，但无绝对标签时会自然塌缩到单一路候选，尤其是 `bbox_center`、`roi_edge_centroid`、`roi_center_patch`、`roi_multi_point` 等。
- `leave_one_weight=0.02/0.05` 能改变被偏好的候选，但不能在无 `known_z` 的单静止片段上解决模型选择问题。
- 明天真实数据到位后，必须至少跑一次 `--stratify-known-z` 的常规 heldout，再跑可选的 `--holdout-known-z <distance>` 未见距离泛化检查。
- 最终优先比较 `calibrated_smoother`、`robust_smooth` 和 `reliability_smoother`；只有 `reliability_smoother` 在 heldout known-distance bias/MAD 和方法审计上同时通过，才考虑作为后续在线轻量头的依据。

## 训练和评估原则

- 不把明显错误的标定或同步数据当成正常噪声学习。
- 多候选深度字段需要记录 invalid reason 或置信度，避免训练时把缺失值误作真实值。
- bbox、circle 和 ROI 几何候选存在强相关，评估时要按组看贡献。
- 在线 100fps 只保留验证过的少量候选；离线可以全量记录用于分析。
- 当前 `legacy_feature_names()` 不把在线 `x/y/z/vx/vy/vz` 状态字段作为训练输入；它读取候选深度、候选统计、质量字段、pair gate 和水印。严格候选-only 可靠性模型仍应保留“不读 `z_stereo/z`”这个约束，并在新增特征时避免把在线 legacy 选择结果重新引入。

## 必备评估指标

- 候选深度有效率。
- 候选间一致性。
- 深度中位数和 MAD。
- 帧间深度抖动。
- 运动模型残差。
- 最终轨迹对落点/实际轨迹的误差。
