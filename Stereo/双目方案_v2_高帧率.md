# Jetson Xavier NX 双目3D定位 — 60-100 FPS 高帧率 Pipeline 方案 (v2)

> **目标**：海康工业相机 + YOLOv8/v10 检测 + 全帧视差深度图 + 3D 坐标输出  
> **平台**：NVIDIA Jetson Xavier NX (JetPack 5.x)  
> **帧率目标**：60-100 FPS (Pipeline 吞吐量)  
> **语言**：C++ (主 Pipeline) / Python (离线标定/调试工具)

---

## 〇、核心设计原则

### 吞吐量 ≠ 延迟之和

原方案将各阶段延迟相加 → ~30ms → 30 FPS，这是**串行思维**。

Pipeline 流水线下：
$$\text{Throughput} = \frac{1}{\max(\text{Stage}_i \text{ latency})}$$

即使总 Latency = 25ms，只要瓶颈 Stage 只有 10ms，吞吐量就是 **100 FPS**。  
代价是引入 N-1 帧的端到端延迟（N = Stage 数量），对于工业检测/定位场景可接受。

## 一、硬件资源盘点 — Xavier NX 计算单元

| 计算单元 | 用途分配 | 说明 |
|:---|:---|:---|
| **GPU** (384 CUDA Cores, 48 Tensor Cores) | 视差计算 (VPI CUDA backend) | Tensor Cores 用于 INT8 推理备选 |
| **NVDLA × 2** | 目标检测 (YOLOv8n/v10n INT8) | 独立于 GPU 的推理引擎 |
| **PVA × 2** | 图像校正 (VPI Remap) | 专用视觉加速器 |
| **CPU** (6-core Carmel ARM) | 图像抓取、Pipeline 调度、坐标转换 | 尽量减少 CPU 参与数据处理 |
| **VIC** (Video Image Compositor) | ISP 处理（如需 Bayer→RGB） | 海康直出 RGB/Mono 时可不用 |

**关键**：NVDLA、PVA、GPU 三者可**同时并行工作**，这是实现高帧率的硬件基础。

## 二、NVDLA 深度分析

### 2.1 NVDLA 是什么

NVDLA (NVIDIA Deep Learning Accelerator) 是 Xavier 系列内嵌的**独立 AI 推理引擎**，不占用 GPU CUDA Cores。Xavier NX 有 **2 个 NVDLA 核心**（DLA_CORE_0、DLA_CORE_1）。

### 2.2 NVDLA 能处理什么

| 支持的操作 | 说明 |
|:---|:---|
| Convolution (Conv2D) | 标准卷积、深度可分离卷积 |
| Deconvolution | 转置卷积 |
| Fully Connected | 全连接层 |
| Activation | ReLU、Sigmoid、Tanh |
| Pooling | Max/Average Pooling |
| BatchNorm | 融合进卷积 |
| Element-wise | Add、Multiply |
| Softmax | 分类输出层 |
| Concatenation | 特征拼接 |

**不支持的操作**（会 fallback 到 GPU）：
- 动态 shape / 动态控制流
- 部分复杂激活函数 (Mish, SiLU 等需要看 TensorRT 版本)
- 某些 Resize/Upsample 操作
- 自定义插件层

### 2.3 YOLOv8n/v10n 在 NVDLA 上的可行性

| 模型 | NVDLA 兼容性 | 说明 |
|:---|:---|:---|
| **YOLOv8n** | ⚠️ 部分层 fallback | SiLU 激活在旧 JetPack 上不被 DLA 支持，会 fallback 到 GPU。可用 `--device DLA` + `--allowGPUFallback` |
| **YOLOv10n** | ⚠️ 同上 | 类似 YOLOv8 的骨干网络结构 |
| **YOLOv8n (ReLU 替换)** | ✅ 几乎全部 on DLA | 将 SiLU 替换为 ReLU，重新训练/微调后，DLA 覆盖率可达 90%+，少量层 fallback GPU |

### 2.4 NVDLA 速度评估

基于 Xavier NX（15W 6-core 模式，`nvpmodel -m 2`）：

| 场景 | 延迟 | 说明 |
|:---|:---|:---|
| YOLOv8n INT8 (纯 GPU) | 8-12ms | 640×640 输入 |
| YOLOv8n INT8 (DLA + GPU fallback) | **10-15ms** | SiLU fallback 到 GPU 增加约 2-3ms 开销 |
| YOLOv8n-ReLU INT8 (纯 DLA) | **12-18ms** | DLA 时钟频率低于 GPU，纯 DLA 略慢 |
| YOLOv8n INT8 (DLA) + 视差 (GPU) **并行** | **DLA 15ms / GPU 10ms → 瓶颈 15ms** | 两者同时工作！ |

**结论**：
- NVDLA **单独速度**略慢于 GPU（DLA 频率低）
- 但 NVDLA 的价值在于**释放 GPU 给视差计算**，两者并行，总吞吐更高
- 如果检测精度允许，用 ReLU 替代 SiLU 重训模型可最大化 DLA 覆盖率
- **推荐策略**：`DLA_CORE_0 + allowGPUFallback`，让 TensorRT 自动调度

## 三、Zero-Copy 零拷贝内存设计

### 3.1 问题

传统流程每帧需要：
```
海康 SDK buffer (CPU) → cudaMemcpyHostToDevice → GPU buffer
```
1280×720 单通道 = 921,600 bytes/帧，双目 = 1.8 MB/帧  
100 FPS → **180 MB/s 拷贝带宽**，PCIe/总线不是瓶颈，但 `cudaMemcpy` 是同步调用，会阻塞 pipeline。

### 3.2 解决方案：Unified Memory / Mapped Pinned Memory

Xavier NX 是 **SoC 统一内存架构**（CPU 和 GPU 共享物理 DRAM），所以零拷贝几乎零开销。

#### 方案一：`cudaHostAllocMapped`（推荐）

```cpp
// 分配 CPU/GPU 共享的 pinned memory
void* h_ptr = nullptr;
cudaHostAlloc(&h_ptr, frameSize, cudaHostAllocMapped);

// 获取 GPU 可访问的指针
void* d_ptr = nullptr;
cudaHostGetDevicePointer(&d_ptr, h_ptr, 0);

// 海康 SDK 直接写入 h_ptr
MV_CC_GetOneFrameTimeout(cameraHandle, h_ptr, frameSize, &frameInfo, 1000);

// GPU kernel 直接从 d_ptr 读取，无需 cudaMemcpy
vpiImageCreateWrapperCudaPitchLinear(/* ... d_ptr ... */);
```

#### 方案二：VPI NvBuffer（VPI 原生零拷贝）

```cpp
// VPI 分配的 buffer 天然支持 CPU/GPU/PVA 共享访问
VPIImage vpiImg;
vpiImageCreate(1280, 720, VPI_IMAGE_FORMAT_U8, VPI_BACKEND_CUDA | VPI_BACKEND_PVA, &vpiImg);

// Lock 获取 CPU 指针，海康 SDK 写入
VPIImageData imgData;
vpiImageLockData(vpiImg, VPI_LOCK_WRITE, VPI_IMAGE_BUFFER_CUDA_PITCH_LINEAR, &imgData);
// 将 imgData.buffer.pitch.planes[0].data 传给海康 SDK
MV_CC_GetOneFrameTimeout(handle, imgData.buffer.pitch.planes[0].data, ...);
vpiImageUnlock(vpiImg);

// 后续 VPI 操作直接使用 vpiImg，无拷贝
vpiSubmitRemap(stream, VPI_BACKEND_PVA, remap, vpiImg, rectifiedImg, ...);
```

### 3.3 内存流转全图

```
┌─────────────────────────────────────────────────────────────┐
│                 Xavier NX 统一物理内存 (LPDDR4x)              │
│                                                             │
│  ┌──────────────┐    零拷贝     ┌──────────────────────┐    │
│  │ 海康 SDK 写入 │ ──────────→ │ VPI/CUDA Buffer      │    │
│  │ (Pinned Mem)  │  同一物理地址  │ GPU/PVA/DLA 直接读取 │    │
│  └──────────────┘              └──────────────────────┘    │
│         │                              │                    │
│         │ CPU 指针                      │ GPU/PVA 指针       │
│         ▼                              ▼                    │
│  ┌──────────┐                  ┌──────────────┐            │
│  │ Pipeline  │                  │ Remap/Stereo │            │
│  │ 调度/坐标 │                  │ /TensorRT    │            │
│  └──────────┘                  └──────────────┘            │
└─────────────────────────────────────────────────────────────┘
```

**关键点**：Xavier NX 的 CPU 和 GPU 共享同一块 DRAM，`cudaHostAllocMapped` 实际上就是映射到同一物理页面，**零开销**。

## 四、Pipeline 架构设计

### 4.1 四级流水线

```
时间轴(ms) →  0    10    20    30    40    50    60    70    80
──────────────────────────────────────────────────────────────
Stage 0 [Grab+Rect]  F0    F1    F2    F3    F4    F5    F6    F7
Stage 1 [Detect/DLA]       F0    F1    F2    F3    F4    F5    F6
Stage 2 [Stereo/GPU]       F0    F1    F2    F3    F4    F5    F6
Stage 3 [Fuse+Output]            F0    F1    F2    F3    F4    F5
                                  ↑     ↑     ↑     ↑     ↑
                               输出开始，每10ms一帧 = 100 FPS
```

**注意**：Stage 1 (DLA) 和 Stage 2 (GPU) **并行执行**，不互相等待。  
它们处理同一帧，Stage 3 等两者都完成后融合。

### 4.2 各 Stage 详细设计

#### Stage 0：图像抓取 + 校正

| 项目 | 说明 |
|:---|:---|
| 输入 | 海康 GPIO 硬触发，100 Hz |
| 操作 | 1) 获取左右帧到 pinned memory buffer; 2) VPI Remap 校正 |
| 硬件 | CPU (抓取) + PVA (校正) |
| 输出 | 校正后的左图 `rectL`、右图 `rectR`（VPIImage） |
| 延迟 | ~3ms (抓取 1ms + 校正 2ms) |

```cpp
// 伪代码
void stage0_grab_and_rectify() {
    // 硬件触发已由外部 GPIO 完成
    MV_CC_GetOneFrameTimeout(camL, bufL.pinnedPtr, ...);
    MV_CC_GetOneFrameTimeout(camR, bufR.pinnedPtr, ...);

    // PVA 加速校正（异步提交）
    vpiSubmitRemap(streamPVA, VPI_BACKEND_PVA, remapL, rawL, rectL, ...);
    vpiSubmitRemap(streamPVA, VPI_BACKEND_PVA, remapR, rawR, rectR, ...);
    vpiStreamSync(streamPVA);
}
```

#### Stage 1：目标检测 (NVDLA)

| 项目 | 说明 |
|:---|:---|
| 输入 | `rectL`（校正后左图） |
| 操作 | YOLOv8n/v10n INT8 推理 |
| 硬件 | **NVDLA Core 0**（少量层 fallback GPU） |
| 输出 | BBox 列表 `detections[]` |
| 延迟 | ~10-15ms |

```cpp
// TensorRT Engine 构建时指定 DLA
config->setDefaultDeviceType(nvinfer1::DeviceType::kDLA);
config->setDLACore(0);
config->setFlag(nvinfer1::BuilderFlag::kGPU_FALLBACK); // 不支持的层回退GPU
config->setFlag(nvinfer1::BuilderFlag::kINT8);

// 推理
context->enqueueV3(dlaStream);
```

#### Stage 2：全帧视差计算 (GPU)

| 项目 | 说明 |
|:---|:---|
| 输入 | `rectL`, `rectR`（校正后左右图） |
| 操作 | VPI Stereo Disparity（全帧） |
| 硬件 | **GPU (CUDA backend)** |
| 输出 | 视差图 `disparityMap` |
| 延迟 | ~8-12ms (见下方优化) |

```cpp
// 使用 CUDA backend 跑全帧视差（PVA 给了 Stage 0）
VPIStereoDisparityEstimatorCreationParams params;
params.maxDisparity = 128;  // 根据基线和场景调整
params.windowSize = 5;

vpiCreateStereoDisparityEstimator(VPI_BACKEND_CUDA, &params, &stereo);
vpiSubmitStereoDisparityEstimator(streamGPU, VPI_BACKEND_CUDA,
    stereo, rectL, rectR, disparityMap, confidenceMap, NULL);
```

**Stage 1 和 Stage 2 在不同硬件上并行执行——这是架构核心。**

#### Stage 3：融合 + 3D 坐标输出

| 项目 | 说明 |
|:---|:---|
| 输入 | `detections[]` + `disparityMap` |
| 操作 | 对每个 BBox 区域提取深度 → 3D 坐标 |
| 硬件 | CUDA kernel (GPU) 或 CPU |
| 输出 | `Object3D[]` (类别, x, y, z, confidence) |
| 延迟 | < 1ms |

```cpp
// CUDA kernel：对每个检测框提取中值视差
__global__ void extractDepthFromBBox(
    const int16_t* disparity, int width,
    const BBox* bboxes, int numBoxes,
    float* depths)
{
    int idx = blockIdx.x;  // 每个 block 处理一个 bbox
    if (idx >= numBoxes) return;

    BBox box = bboxes[idx];
    // 收集框内视差值，用直方图法求峰值
    // ... (共享内存直方图)
    float d_peak = computeHistogramPeak(disparity, box, width);
    depths[idx] = (focal * baseline) / d_peak;
}

// 3D 投影
for (auto& det : detections) {
    det.Z = depths[det.id];
    det.X = (det.cx - cx) * det.Z / focal;
    det.Y = (det.cy - cy) * det.Z / focal;
}
```

### 4.3 Pipeline 同步机制

```cpp
// 使用 CUDA Events 实现 Stage 间同步
// Ring Buffer：3 组帧 buffer 做三缓冲

struct FrameSlot {
    VPIImage rawL, rawR;        // pinned + mapped
    VPIImage rectL, rectR;      // 校正后
    VPIImage disparity;         // 视差图
    cudaEvent_t evtRectDone;    // Stage 0 完成
    cudaEvent_t evtDetectDone;  // Stage 1 完成
    cudaEvent_t evtStereoDone;  // Stage 2 完成
    std::vector<Detection> dets;
    std::vector<Object3D> results;
};

// 三缓冲环形队列
FrameSlot slots[3];
int grabIdx = 0, detectIdx = 0, stereoIdx = 0, fuseIdx = 0;

void pipelineLoop() {
    while (running) {
        auto& s = slots[grabIdx % 3];

        // Stage 0: Grab + Rectify
        stage0_grab_and_rectify(s);
        cudaEventRecord(s.evtRectDone, streamPVA);
        grabIdx++;

        // Stage 1 & 2: 等校正完成后并行启动
        if (grabIdx > 0) {
            auto& s1 = slots[detectIdx % 3];
            cudaStreamWaitEvent(dlaCudaStream, s1.evtRectDone, 0);
            stage1_detect(s1);  // DLA 异步
            cudaEventRecord(s1.evtDetectDone, dlaCudaStream);

            cudaStreamWaitEvent(gpuStream, s1.evtRectDone, 0);
            stage2_stereo(s1);  // GPU 异步
            cudaEventRecord(s1.evtStereoDone, gpuStream);
            detectIdx++; stereoIdx++;
        }

        // Stage 3: 等 Stage 1 & 2 都完成
        if (detectIdx > 0 && stereoIdx > 0) {
            auto& s3 = slots[fuseIdx % 3];
            cudaStreamWaitEvent(fuseStream, s3.evtDetectDone, 0);
            cudaStreamWaitEvent(fuseStream, s3.evtStereoDone, 0);
            stage3_fuse(s3);
            fuseIdx++;
            outputResult(s3.results);  // 输出 3D 坐标
        }
    }
}
```

## 五、视差帧率优化策略

全帧视差是本方案的默认模式，但如果帧率不达标，保留以下降级策略：

### 策略 A：ROI 视差（仅计算检测框区域）

适用于"只关心目标深度、不需要全场深度图"的场景。

```
默认模式：全帧 1280×720 视差 → ~10-12ms (GPU)
策略 A ：仅对检测框区域计算视差
         假设 5 个目标、平均框大小 200×200
         总面积 = 200,000 pixels vs 全帧 921,600 pixels → 约 1/4.6
         预期延迟：3-4ms
```

**实现方式**：
```cpp
// 对每个检测框裁剪左右图 ROI，单独提交 VPI 视差
for (auto& det : detections) {
    VPIImage roiL = cropROI(rectL, det.bbox, margin=32); // 左右各扩 32px
    VPIImage roiR = cropROI(rectR, det.bbox_shifted, margin=32);
    vpiSubmitStereoDisparityEstimator(stream, VPI_BACKEND_CUDA,
        stereo, roiL, roiR, roiDisparity, ...);
}
```

**注意**：策略 A 要求检测先于视差完成（Pipeline 改为串行依赖），会增加单帧延迟但减少计算量。

### 策略 B：降分辨率视差

适用于"需要全场深度图但可接受精度损失"的场景。

```
默认模式：1280×720 → ~10-12ms
策略 B ：640×360  → ~3-5ms
         480×270  → ~2-3ms
```

**精度影响**（基线 120mm 为例）：

| 分辨率 | 5m 处深度分辨率 | 适用场景 |
|:---|:---|:---|
| 1280×720 | ±2cm | 精密定位 |
| 640×360 | ±5cm | 工业巡检/避障 |
| 480×270 | ±8cm | 粗定位/计数 |

**实现方式**：
```cpp
// VPI Rescale 硬件加速降采样
VPIImage halfL, halfR, halfDisp;
vpiImageCreate(640, 360, VPI_IMAGE_FORMAT_U8, 0, &halfL);
vpiSubmitRescale(streamGPU, VPI_BACKEND_CUDA, rectL, halfL, VPI_INTERP_LINEAR, ...);
vpiSubmitRescale(streamGPU, VPI_BACKEND_CUDA, rectR, halfR, VPI_INTERP_LINEAR, ...);
vpiSubmitStereoDisparityEstimator(streamGPU, VPI_BACKEND_CUDA,
    stereo, halfL, halfR, halfDisp, ...);
// 视差值需要 ×2 还原到原始分辨率尺度
```

### 策略选择决策表

| 帧率目标 | 检测 (DLA) | 视差 (GPU) | 吞吐瓶颈 | 达标？ |
|:---|:---|:---|:---|:---|
| **60 FPS** | 15ms | 12ms (全帧 1280×720) | 15ms → 66 FPS | ✅ |
| **80 FPS** | 15ms | 5ms (策略 B: 640×360) | 15ms → 66 FPS | ❌ DLA 是瓶颈 |
| **80 FPS** | 10ms (ReLU模型/GPU) | 5ms (策略 B) | 10ms → 100 FPS | ✅ |
| **100 FPS** | 10ms (GPU) | 12ms (全帧) | 12ms → 83 FPS | ❌ |
| **100 FPS** | 10ms (GPU) | 5ms (策略 B) | 10ms → 100 FPS | ✅ |
| **100 FPS** | 8ms (GPU INT8) | 3ms (策略 A: ROI) | 8ms → 125 FPS | ✅ |

**结论**：
- **60 FPS**：默认方案（DLA检测 + GPU全帧视差）即可达成
- **80-100 FPS**：需要策略 B（降分辨率）或回退到 GPU 检测 + 策略 A/B
- **100 FPS+**：检测必须在 GPU 上跑（8ms），视差用策略 A 或 B

## 八、性能剖析与调优

### 必须使用 Nsight Systems 确认时序

```bash
# 录制 5 秒 profiling
nsys profile -t cuda,nvtx,osrt -o pipeline_trace ./stereo_pipeline

# 在主机上打开 .nsys-rep 文件查看 timeline
# 确认：DLA 和 GPU stream 是否真正并行
# 确认：是否有意外的 cudaMemcpy / cudaDeviceSynchronize 阻塞
```

### 关键指标检查清单

| 指标 | 达标值 | 不达标怎么办 |
|:---|:---|:---|
| Pipeline 吞吐 | ≥ 60 FPS | 检查瓶颈 Stage，启用策略 B |
| cudaMemcpy 调用数 | **0** | 改用零拷贝 |
| CPU 占用率 | < 30% | 检查是否有 OpenCV CPU 操作泄漏 |
| GPU 利用率 | > 70% | 若太低说明 pipeline stall |
| DLA 利用率 | > 60% | 若太低说明 fallback 层太多 |
| 端到端延迟 | < 50ms | Pipeline 深度 × Stage 延迟 |

## 九、完整延迟/吞吐预算

### 默认模式（全帧视差，DLA 检测）→ 目标 60-66 FPS

| Stage | 模块 | 硬件 | 延迟 | 并行关系 |
|:---|:---|:---|:---|:---|
| 0 | Grab + Rectify | CPU + PVA | 3ms | - |
| 1 | YOLOv8n INT8 | **NVDLA** | 12-15ms | ↕ 并行 |
| 2 | VPI Stereo (全帧 1280×720) | **GPU** | 10-12ms | ↕ 并行 |
| 3 | Depth Extract + 3D | GPU/CPU | < 1ms | - |
| **Pipeline 瓶颈** | | | **max(15, 12) = 15ms** | |
| **吞吐量** | | | **66 FPS** | ✅ |
| **端到端延迟** | | | **3+15+1 = 19ms** | |

### 激进模式（降分辨率视差，GPU 检测）→ 目标 100+ FPS

| Stage | 模块 | 硬件 | 延迟 | 并行关系 |
|:---|:---|:---|:---|:---|
| 0 | Grab + Rectify | CPU + PVA | 3ms | - |
| 1 | YOLOv8n INT8 | **GPU (Stream A)** | 8-10ms | ↕ 时分复用 |
| 2 | VPI Stereo (640×360) | **GPU (Stream B)** | 3-5ms | ↕ 时分复用 |
| 3 | Depth Extract + 3D | CPU | < 1ms | - |
| **Pipeline 瓶颈** | | | **~10ms** | |
| **吞吐量** | | | **100 FPS** | ✅ |
